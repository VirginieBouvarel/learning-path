Projet fil rouge

KataSensei

Une plateforme d'entraînement aux katas de code pour les profils en reconversion — bootcamp, formation courte. Un éditeur Monaco embarqué, une IA qui joue le rôle d'un senior bienveillant (hints graduels, feedback sur le code, jamais la réponse directe), un compte utilisateur avec suivi de progression. L'exécution du code passe par Piston API (sandbox gratuite, multi-langages). Backend Java 21 + Spring Boot, archi hexagonale, frontend Vue 3 / TypeScript.

Vue 3 + TypeScript Monaco Editor Java 21 + Spring Boot 3 PostgreSQL + Flyway Piston API (exécution) Claude API (IA sensei) JWT Auth Architecture hexagonale Docker + CI

🔌Spring Boot = event listener HTTP≈ v-on côté serveur

@RestController + @GetMapping. En Vue tu écoutes des clics. En Java, tu écoutes des requêtes HTTP. Tu écris le test JUnit avant le handler — ton premier cycle Red/Green.

🔄TDD Red/Green/Refactor — cycle de base≈ écrire le type TS avant l'implémentation

Pour chaque endpoint : 1) test qui échoue (rouge) → 2) code minimal pour le faire passer (vert) → 3) refacto sans casser le test. Le rythme du reste du parcours.

🗄JPA naïf + Flyway migrations≈ Prisma mais sans architecture

Intentionnellement naïf : logique dans le contrôleur, pas de use case. Flyway versionne le schéma comme git versionne le code.

🚨Gestion d'erreurs centralisée — @ControllerAdvice≈ error boundary global

Un seul endroit pour toutes les erreurs. Problem Details RFC 7807 : ton front Vue reçoit toujours le même format JSON structuré. Hiérarchie DomainException — dès la phase 1.

KATASENSEI À CETTE ÉTAPE

GET /katas et POST /katas persistant en PostgreSQL. Le frontend Vue affiche la liste des katas. Toutes les erreurs arrivent en JSON structuré (jamais un stack trace HTML).

LIVRABLE TECHNIQUE

CRUD Kata en TDD. @ControllerAdvice global. Problem Details RFC 7807. CI enrichie avec les nouveaux tests.

›
🏷️Branded types : KataId, UserId≈ newtype en Haskell/Rust

Impossible de mélanger les IDs. Le compilateur TS devient ton premier testeur. KataId et UserId ne sont pas interchangeables.

🔀Discriminated unions — états de session≈ sealed classes Java

SessionState : idle | running | completed. Le compilateur interdit de lire score si status !== 'completed'.

✅Result pattern + type guards≈ Either en FP

Result<T, E> : l'appelant est forcé de gérer les deux cas (ok / error). Plus de try/catch oubliés.

🪵LogService typé — adapter sortant front≈ AiSensei côté front

Interface LogPort. ConsoleLogger en dev, SentryLogger en prod. Ton premier adapter sortant front — même principe que les ports hexa.

⏳useAsync — loader headless réutilisable≈ renderless component

const { data, status, error } = useAsync(() => kataRepo.findById(id)). Réutilisé sur toutes les pages de KataSensei.

🔷Archi hexa front — domain/infra/ui≈ separation of concerns Vue avancée

domain/ = types purs, use cases TS, zéro import framework. infrastructure/ = ApiKataRepository. ui/ = composants Vue via inject.

KATASENSEI À CETTE ÉTAPE

Front Vue typé à fond. Impossible de rendre un résultat si la session n'est pas running. Impossible de mélanger KataId et UserId. LogService actif. useAsync sur toutes les pages.

LIVRABLE TECHNIQUE

Branded types + Result + useAsync. Refactoring front phase 1. Tests Vitest des type guards. Archi domain/infra/ui en place.

🛡️Spring Security + JWT≈ axios interceptor

Spring intercepte chaque requête avant ton code. Token invalide → 401 automatique. TDD : tu écris d'abord le test "accès refusé sans token" avant d'implémenter le filtre.

🏷️Value Objects Java — miroir des branded types≈ branded types TS

record UserId(UUID value). Le compilateur Java refuse KataId là où UserId est attendu. Même garantie que les branded types TS.

🧱SOLID dans KataSensei — pas dans un livre≈ principes appliqués au vrai code

Chaque principe illustré par mauvaise version → version corrigée dans le code de KataSensei. Le diff est la leçon.

🚨AppError union front + hiérarchie back≈ discriminated union d'erreurs exhaustive

Front : le compilateur force le traitement de chaque cas (network, validation, not-found, unauthorized). Back : sealed class DomainException.

KATASENSEI À CETTE ÉTAPE

Auth JWT end-to-end. Un utilisateur s'inscrit, se connecte, voit ses katas. Les Value Objects Java mirroient les branded types TS. Gestion d'erreurs exhaustive des deux côtés.

LIVRABLE TECHNIQUE

Auth complète en TDD. AppError union. Refactoring back phase 1. ADR 002 écrit.

🔷Hexagonale back — le domaine ne dépend de rien≈ composable pur sans import axios

domain/ = zéro import Spring. model/, port/in/, port/out/, usecase/. Tests unitaires domaine < 50ms. Pas de DB, pas de réseau.

🔸Hexagonale front — miroir du back≈ archi hexa côté Vue

domain/ = types purs, use cases TS, zéro import Vue. infrastructure/ = adapters. ui/ = composants via provide/inject. Testable sans navigateur.

📦DDD tactique — Aggregates + Domain Events≈ invariants métier garantis

KataSession = aggregate root. On ne modifie jamais ses entités internes directement. Domain Events : KataCompleted, HintRequested, StreakBroken.

🎯Strategy / Factory / Builder≈ design patterns appliqués

Strategy : HintStrategy swappable. Factory : AiSenseiFactory.create(config). Builder : KataSession.builder().withKata().withUser().build().

🧩Compound Component + Provider pattern≈ composition Vue avancée

<KataCard> <KataCard.Header /> <KataCard.Body /> </KataCard>. provide/inject = miroir exact du DI Java.

KATASENSEI À CETTE ÉTAPE

L'IA entre en scène. L'utilisateur choisit un kata, écrit du code dans Monaco, l'exécute via Piston API, demande un hint au sensei IA. Le sensei répond de façon socratique.

LIVRABLE TECHNIQUE

Hexa front + back opérationnels. POST /sessions/{id}/run + /hint. Tests domaine < 50ms. Design patterns documentés. ADR 003.

👁️Observer / Spring Events — Domain Events≈ EventBus mitt en Vue

KataCompleted publié par le use case. StreakListener, BadgeListener, ProgressionListener écoutent indépendamment. Le domaine ne sait pas combien de listeners existent.

🌊SSE streaming — useHintStream headless≈ EventSource côté Vue

Plutôt qu'attendre 4–5s la réponse de Claude complète, tu streames token par token. SseEmitter Java + EventSource Vue. Composable headless réutilisable.

🔄Redis cache @Cacheable≈ computed Vue

@Cacheable('katas') : lue une fois en DB, servie depuis Redis ensuite. @CacheEvict quand un kata est modifié. Comme un computed qui recalcule seulement quand les dépendances changent.

🛡️Anti-Corruption Layer front≈ mapper domaine / API

Dans infrastructure/ — jamais dans domain/. Le domaine front ne parle jamais le langage de l'API externe. Mapping ApiKataDto → Kata.

KATASENSEI À CETTE ÉTAPE

Dashboard de progression (katas complétés, streak, niveau préféré). Le hint IA s'affiche token par token en temps réel. Les badges se débloquent sans que le use case ne sache qu'ils existent.

LIVRABLE TECHNIQUE

GET /users/me/progress. SSE /hint/stream. Events découplés. Cache Redis. ACL front. Tests des listeners.

📜OpenAPI contract-first + client TS généré≈ tRPC sans lock-in

Le YAML OpenAPI est la loi. openapi-generator génère les types TS et les fonctions fetch pour Vue. Si le back change un champ sans mettre à jour la spec, la CI casse.

🧩Decorator pattern — logging et métriques≈ higher-order function TS

LoggedKataRepository wrape sans modifier le comportement. Le domaine ne sait rien. Composition > héritage.

🧪Testcontainers — vraie DB en CI≈ tests avec vraie infra

Vrai PostgreSQL + Redis dans Docker pendant les tests d'intégration. Fini les mocks de DB qui ne ressemblent pas à la prod.

🎭Playwright — tests e2e bout en bout≈ tests utilisateur réels

L'utilisateur peut compléter un kata : navigation, écriture de code, exécution, vérification du résultat. Scénarios bout en bout.

KATASENSEI À CETTE ÉTAPE

KataSensei a un contrat d'API vérifié en CI. Les tests d'intégration tournent contre une vraie DB. Les tests Playwright valident les parcours utilisateur clés. Sentry capture les erreurs en prod.

LIVRABLE TECHNIQUE

Client TS + ACL. Testcontainers verts. Playwright. Sentry. ADR archivés. README final.

🐳Docker multi-stage — JDK → JRE léger≈ npm build optimisé

Stage 1 : build (JDK complet). Stage 2 : run (JRE léger, 80% plus petit). Le même Dockerfile pour dev et prod.

⚙️Pipeline GitHub Actions complet — 5 étapes≈ CI/CD complète

unit-tests → integration-tests → e2e-tests → build-docker → deploy. Si une étape échoue, le déploiement ne part pas.

🏥Health checks + Actuator + graceful shutdown≈ DevTools Vue en prod

KataSensei est en prod. Un push sur main déclenche le pipeline complet. En cas d'échec, le déploiement ne part pas. Les reconversions peuvent s'inscrire et commencer leurs katas.

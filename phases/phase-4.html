Phase 4

Architecture hexagonale complète + DDD + patterns front

Durée estimée : ~14h réparties sur 3 semaines

La phase 4 est le grand basculement du projet : tu prends le code utile mais encore naïf des phases 1 à 3 et tu le restructures pour que le domaine soit testable, lisible et autonome des frameworks. C’est aussi l’entrée réelle de l’IA dans KataSensei avec les adapters Piston et Claude.

1

Ce que tu reconstruis vraiment

Jusqu’ici, le projet prouve que les fonctionnalités existent. À partir d’ici, tu rends le système durable. Le but n’est pas de “faire de l’archi pour faire joli”, mais de faire en sorte qu’un changement métier n’oblige plus à traverser Spring, JPA, Vue, l’API IA et l’exécution sandbox en même temps.

Point de méthode Cette phase se mène au compilateur et aux tests. Tu ne réécris pas tout. Tu déplaces, tu extrais, tu renomme, puis tu remets les adapters autour du domaine.
2

Les 4 chantiers de la phase

2.1 — Hexagonale back

Créer la séparation domain/port/usecase/infrastructure côté Spring Boot. Le domaine ne doit plus importer Spring, JPA ou des clients HTTP.

2.2 — Hexagonale front

Faire miroir côté Vue avec domain/, infrastructure/ et ui/. Le composant n’appelle pas l’API directement ; il dépend d’un port injecté.

2.3 — DDD tactique et patterns

Modéliser KataSession comme aggregate root, nommer les événements métier, et utiliser Strategy / Factory / Builder là où ils clarifient une variation réelle du système.

2.4 — Adapters externes

Introduire proprement l’exécution de code via Piston et les hints IA via Claude. Le domaine doit exprimer une intention, pas connaître la forme des APIs externes.

3

Ordre recommandé

  1. Figer l’existant avec les tests utiles des phases 1 à 3.
  2. Créer les packages et déplacer d’abord le domaine backend sans changer les comportements.
  3. Extraire les ports entrants et sortants, puis rebrancher les adapters Spring/JPA.
  4. Faire le même travail côté frontend avec les repositories API et les composants Vue.
  5. Introduire KataSession, puis brancher Piston et Claude derrière des ports dédiés.
  6. Finir par les patterns utiles et l’ADR qui explique les choix faits.
Important Si tu bloques entre “déplacer des fichiers” et “repenser le modèle”, sépare les deux. Le déplacement structurel doit rester mécanique ; la refonte métier vient ensuite.

GUIDE DÉTAILLÉ À VENIR

Cette page sert d’overview opérationnelle. Le guide pas-à-pas détaillé de la phase 4 sera rédigé avant l’implémentation complète.

☑ Checklist de phase

  • Le backend a une structure domain / port / usecase / infrastructure.
  • Le frontend a un découpage domain / infrastructure / ui cohérent avec le back.
  • KataSession porte les invariants métier importants.
  • Piston et Claude sont branchés via des ports sortants testables.
  • Les tests domaine back restent rapides et n’ont ni DB ni réseau.
  • L’ADR de la phase explique les choix et leurs conséquences.

Ce que tu dois sentir à la sortie

  • Tu sais distinguer une règle métier d’un détail d’infrastructure.
  • Tu sais brancher une API externe sans contaminer le domaine.
  • Tu sais justifier quand un pattern clarifie le code, et quand il ajoute juste du bruit.

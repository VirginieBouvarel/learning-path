Phase 5

Events, SSE streaming, cache Redis + ACL front

Durée estimée : ~10h réparties sur 2 semaines

La phase 5 rend KataSensei vivant. Les événements métier déclenchent des réactions découplées, le hint IA devient progressif grâce au streaming, les lectures répétitives passent par Redis et le front protège son domaine via une anti-corruption layer claire.

Ce qui change dans l’expérience utilisateur

Le système ne répond plus seulement “une fois terminé”. Il commence à réagir au fil de l’eau. Lorsqu’un kata est complété, plusieurs conséquences peuvent suivre. Lorsqu’un hint est demandé, la réponse apparaît token par token. Lorsqu’une liste est lue souvent, Redis amortit la charge.

2

Les chantiers de la phase

2.1 — Domain Events et listeners

Créer des faits métier nommés proprement comme KataCompleted et les publier depuis les use cases. Chaque réaction annexe doit vivre dans son listener dédié.

2.2 — Streaming de hints

Mettre en place SseEmitter côté Spring et un composable type useHintStream côté Vue. L’objectif est de traiter le flux comme une primitive réutilisable, pas comme du code ad hoc collé dans un composant.

2.3 — Cache Redis

Choisir un ou deux cas de lecture vraiment répétitifs, puis mesurer la cohérence du cache avec les invalidations associées. Le cache doit accélérer, pas introduire des bugs fantômes.

2.4 — Anti-Corruption Layer front

Formaliser les mappers d’API dans l’infrastructure front pour que le domaine ne parle jamais le dialecte du backend ni des APIs externes.

3

Ordre recommandé

  1. Commencer par les Domain Events pour clarifier les responsabilités métier.
  2. Brancher les listeners les plus simples, puis valider leurs tests indépendants.
  3. Ajouter ensuite le streaming SSE de bout en bout avec un seul flux de hints.
  4. Introduire Redis sur une lecture stable et tracer précisément les invalidations.
  5. Finir par l’ACL front pour consolider les contrats manipulés côté Vue.
Vigilance La tentation est forte de cacher trop tôt ou de publier trop d’événements. N’ajoute un event ou un cache que si la responsabilité ou le coût qu’il traite est déjà tangible.

GUIDE DÉTAILLÉ À VENIR

Cette page sert d’overview opérationnelle. Le guide pas-à-pas détaillé de la phase 5 sera rédigé avant l’implémentation complète.

☑ Checklist de phase

  • Les événements métier sont nommés avec le vocabulaire du domaine.
  • Les listeners sont indépendants et testés séparément.
  • Le hint stream fonctionne de bout en bout via SSE.
  • Le cache Redis couvre un vrai cas d’usage et ses invalidations.
  • Le front mappe les DTOs API vers des objets domaine internes.

Ce que tu dois sentir à la sortie

  • Tu sais découpler un use case de ses effets secondaires.
  • Tu sais raisonner sur le temps réel côté produit sans perdre la maîtrise du code.
  • Tu sais où placer un mapper pour protéger le domaine front.

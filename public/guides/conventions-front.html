ADR 003 - Conventions front (CSS, composants, structure)

Décisions techniques structurantes pour le front KataSensei. Écrit une fois, stable.

Contexte

Le front Vue 3 de KataSensei nécessite des choix explicites sur le CSS, les composants, et la structure des fichiers de test. Ces choix affectent chaque composant du projet — ils doivent être décidés avant d'écrire le premier composant.

1

CSS — composants métier en <style scoped>

Les composants métier KataSensei peuvent utiliser <style scoped> pour isoler leur style local. Quand un composant métier a besoin de classes propres, la convention de nommage suit BEM (Block Element Modifier).

/* Block */
.kata-card { ... }

/* Element */
.kata-card__titre { ... }
.kata-card__niveau { ... }

/* Modifier */
.kata-card--selectionne { ... }
.kata-card__niveau--difficile { ... }
ℹ️ Pourquoi scoped et pas CSS Modules ? CSS Modules nécessite class="$style.titre" dans les templates — plus verbeux sans gain réel en SFC. scoped reste une bonne convention pour les styles locaux de composants métier.
ℹ️ Comment concilier cela avec PrimeVue ? `PrimeVue` fournit des composants déjà stylés. Les composants métier KataSensei restent des composants de projet : ils assemblent les briques PrimeVue, gardent leur responsabilité métier et peuvent ajouter un style local en CSS natif ou SCSS quand c'est utile.
2

Composants — PrimeVue

On utilise PrimeVue comme bibliothèque de composants Vue prête à l'emploi. Le projet garde ses composants métier au-dessus de cette couche et peut les styliser en CSS natif ou en SCSS sans dépendre de Tailwind.

ℹ️ Pourquoi pas Vuetify ? Vuetify impose une identité Material plus marquée. `PrimeVue` reste plus neutre visuellement, plus simple à intégrer dans un produit qui veut garder sa propre direction UI.
3

SFC entiers — pas de découpage

Chaque composant Vue est un fichier .vue unique contenant <template>, <script setup>, et <style scoped>. Pas de découpage en fichiers .html + .css + .ts séparés.

ℹ️ Pourquoi ? Vite, Volar et Vitest sont optimisés pour les SFC monofichiers. Le découpage à la Angular ajoute de la complexité de navigation sans bénéfice dans l'écosystème Vue.
4

Structure des tests — dossier tests/ séparé

Les tests sont dans un dossier tests/ à côté de src/, reproduisant la même arborescence.

katasensei-front/
  src/
    domain/
      usecases/
        GetKatasUseCase.ts
    ui/
      components/
        KataCard.vue
  tests/
    domain/
      usecases/
        GetKatasUseCase.test.ts
    ui/
      components/
        KataCard.test.ts
ℹ️ Pourquoi pas de dossier par composant ? Un dossier KataCard/ contenant KataCard.vue + KataCard.test.ts crée une navigation plus lourde pour seulement 2 fichiers. Le dossier tests/ séparé donne une vue claire de ce qui est testé.
5

Linter CSS — Stylelint

Stylelint est configuré à partir de la phase 3 avec une base standard solide et une syntaxe compatible avec les SFC Vue. Il sert à garder des styles durables et cohérents quand les composants métier commencent à se multiplier.

Alternatives rejetées

AlternativeRaison du rejet
Vuetify 3Design system opinionnaire, lock-in fort
CSS ModulesPlus verbeux que scoped sans gain réel en SFC
Dossier par composantNavigation lourde pour 2 fichiers par composant
Colocalization testsMélange code source et tests dans le même dossier

Conséquences

  • Les composants métier peuvent utiliser <style scoped> et une convention de classes lisible
  • `PrimeVue` est introduit en phase 3, au moment où le catalogue reçoit ses vrais composants UI métier
  • Stylelint est configuré en phase 3, au moment où la dette CSS devient réellement structurante
  • Les tests vivent dans tests/ avec la même arborescence que src/
  • Les composants `PrimeVue` restent une base technique, distincte des composants métier du domaine

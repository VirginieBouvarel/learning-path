Phase 2

Architecture front typée pour sécuriser le catalogue de katas

Durée estimée : ~10h à ~12h réparties sur 2 semaines

La phase 1 a livré un premier catalogue de katas visible côté frontend, mais le flux reste encore naïf : données HTTP trop proches de l'UI, logique de chargement dispersée, types trop larges, et tests front encore peu structurés. La phase 2 refactorise ce flux de bout en bout pour obtenir un frontend Vue plus robuste, typé, testable et prêt à accueillir de vrais composants métier dans la phase suivante.

🟣Typage fort 🟢Hexagone front 🔵Tests

Objectif de phase

Refactoriser le flux de lecture des katas côté frontend pour passer d'un branchement direct UI ↔ HTTP à une architecture claire : types métier, mapper, repository, use case, store, composable UI et vue refactorisée.

État de départ exact

  • Tu travailles dans Ubuntu via WSL2 sur Windows 11.
  • Le repo KataSensei existe déjà et la phase 1 est terminée.
  • Le backend expose déjà au minimum `GET /katas` et `GET /katas/{id}`.
  • Le frontend affiche déjà un catalogue de katas, mais avec une structure encore simple et peu découplée.
  • Le projet utilise déjà Vue 3, TypeScript strict, Pinia et Vitest.

Résultat attendu en fin de phase

  • Le frontend possède une structure `domain / infrastructure / stores / ui` explicite.
  • Les types métier utiles au catalogue sont introduits côté front.
  • Le contrat HTTP est isolé dans un DTO et un mapper.
  • Le chargement des katas passe par un `KataRepository`, un `GetKatasUseCase` et un store Pinia.
  • La vue de liste n'appelle plus `fetch` directement.
  • La structure miroir de `tests/` est en place pour les couches front refactorisées.
  • `npm run type-check` et les tests ciblés passent.

Valeur livrée à l'utilisateur

Le catalogue continue à fonctionner, mais il repose désormais sur une base front beaucoup plus fiable, ce qui réduit le risque de régression et prépare la phase 3, centrée sur les composants UI métier avec `PrimeVue`.

Prérequis

  • La phase 1 est terminée et le catalogue de katas s'affiche encore côté frontend.
  • Tu sais relancer le backend et le frontend en local.
  • Tu sais lancer les tests unitaires front hérités des phases précédentes.

Outils mobilisés

  • Terminal Ubuntu dans WSL
  • Vue 3
  • TypeScript strict
  • Pinia
  • Vitest
  • Fetch API
  • Documentation interne du projet et `ADR 001`
Scripts frontend attendus à partir de cette phase
Le frontend KataSensei doit exposer au minimum les scripts suivants dans package.json. Si type-check n'existe pas encore dans ton projet, ajoute-le maintenant avec exactement cette valeur. 
{
  "scripts": {
    "dev": "vite",

Info parcours
On ne construit pas encore les composants UI métier avec `PrimeVue`. Ici, on stabilise d'abord le flux de données et la structure front. `PrimeVue`, `Stylelint` et `LogService` arriveront dans la phase suivante.
Prérequis technique — à faire avant de commencer

Si Pinia n'a pas été installé lors de la création du projet Vue en phase 0, installe-le maintenant.

Ouvre un terminal Ubuntu dans WSL, place-toi dans le dossier frontend, puis lance :

cd ~/workspace/katasensei/katasensei-front
npm install pinia

Ouvre ensuite src/main.ts et vérifie qu'il contient bien ces deux lignes. Si elles sont absentes, ajoute-les :

import { createPinia } from 'pinia'
// ...
app.use(createPinia())

Résultat attendu : npm run dev démarre sans erreur liée à Pinia.

Alias @/ — configuration obligatoire

Tous les imports de cette phase utilisent l'alias @/ pour pointer vers src/. Il doit être configuré dans deux fichiers avant de créer les premiers fichiers métier.

Ouvre vite.config.ts et ajoute la section resolve.alias :

import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { fileURLToPath, URL } from 'node:url'

export default defineConfig({
  plugins: [vue()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./src', import.meta.url))
    }
  }
})

Ouvre ensuite tsconfig.app.json (ou tsconfig.json si tu n'as qu'un seul fichier tsconfig) et vérifie que compilerOptions contient :

{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}

Résultat attendu : npm run type-check ne signale plus d'erreur de module introuvable sur les imports @/.

1

Vérifier le point de départ front et repérer les fichiers encore naïfs

Objectif
Constater précisément comment le catalogue est branché aujourd'hui avant de commencer le refactoring.

Pourquoi cette étape existe maintenant
Une architecture utile doit résoudre une douleur réelle. Si tu ne localises pas les points faibles de la version phase 1, tu risques d'introduire des couches abstraites sans comprendre ce qu'elles améliorent.

État de départ exact
Le backend et le frontend existent déjà. Le catalogue fonctionne, mais sa structure interne peut encore mélanger UI, accès HTTP et gestion d'état.

Étapes

  1. Ouvre un terminal Ubuntu dans WSL. Résultat attendu : tu travailles bien dans un shell Linux et non dans PowerShell.
  2. Place-toi dans le frontend du projet, puis vérifie ton dossier courant : 
    cd ~/workspace/katasensei/katasensei-front
pwd
ls
    Résultat attendu : `pwd` affiche le dossier du frontend et `ls` montre au minimum `src`, `package.json` et éventuellement `tests`.
  3. Observe la structure actuelle du code et des tests : 
    find src -maxdepth 2 -type f | sort
find tests -maxdepth 2 -type f | sort
    Résultat attendu : tu vois les dossiers et fichiers déjà présents. Si `find tests` affiche `No such file or directory`, note-le : cela confirme que la structure miroir de `tests/` fait bien partie du travail de cette phase.
  4. Repère les appels HTTP directs et les états encore dispersés : 
    rg -n "fetch\\(" src
rg -n "isLoading|loading|error" src
    Résultat attendu : tu identifies précisément quels fichiers parlent encore au réseau ou gèrent eux-mêmes l'état de chargement. Ouvre ensuite au moins un fichier trouvé dans l'éditeur et vérifie si la même unité de code fait à la fois appel HTTP, transformation de données et rendu UI.
  5. Relance le frontend pour vérifier que le point de départ est encore fonctionnel : 
    npm run dev
    Résultat attendu : l'application est accessible et le catalogue s'affiche encore dans le navigateur. Si la commande échoue, corrige d'abord ce problème avant de refactoriser.

Erreurs fréquentes

  • Si `find tests` échoue, ne panique pas : cela veut dire que le dossier n'existe pas encore, pas forcément que le projet est cassé.
  • Si `rg` ne trouve aucun `fetch`, cherche aussi `axios` ou une abstraction réseau déjà existante ; l'objectif est de comprendre le flux réel.
  • Si le frontend ne démarre plus, corrige d'abord ce problème avant tout refactoring.

Exercice 1.1

Écris une phrase qui commence par Le problème actuel est que... et qui décrit pourquoi la vue de liste est encore trop couplée.

Solution - A consulter après 20 min

Une bonne réponse ressemble à : Le problème actuel est que la vue de liste connaît trop de détails techniques, comme le chargement HTTP et la forme brute des données, ce qui la rend fragile dès qu'un contrat API ou un état de chargement change.

Exercice 1.2

Dans tes notes locales, écris noir sur blanc trois douleurs du code actuel. Tu dois nommer au moins :

  • un problème de typage ;
  • un problème de structure ;
  • un problème de testabilité.
Solution - A consulter après 20 min

Exemple de réponse attendue :

  • Typage : les réponses API circulent encore sous forme d'objets trop larges ou de `string` brutes.
  • Structure : la vue ou le composable connaissent encore l'URL de l'API, le `fetch` et la transformation des données.
  • Testabilité : il est difficile de tester le chargement du catalogue sans lancer Vue ou sans simuler directement le réseau.

Questions théoriques

1. Pourquoi démarrer une phase de refactoring par une observation de l'existant ?

Réponse attendue

Parce qu'un refactoring répond à des problèmes déjà présents. Sans diagnostic concret, on risque de déplacer du code sans améliorer sa lisibilité ou sa robustesse.

2. Pourquoi un `fetch` direct dans une vue devient-il vite coûteux ?

Réponse attendue

Parce que la vue doit alors gérer à la fois l'affichage, le chargement, les erreurs réseau et parfois la transformation des données. Elle cumule trop de responsabilités.

3. Pourquoi une architecture front doit-elle rester liée à une fonctionnalité visible ?

Réponse attendue

Parce que l'objectif n'est pas de produire des couches abstraites, mais de rendre une fonctionnalité plus sûre, plus testable et plus facile à faire évoluer.

Checklist manuelle

  • J'ai relancé le frontend et le catalogue s'affiche.
  • J'ai repéré les appels HTTP directs encore présents.
  • J'ai noté les douleurs principales du code actuel.

2

Poser la structure cible et la structure miroir de tests

Objectif
Créer une arborescence claire pour le code front et ses tests avant d'y déplacer la logique.

Pourquoi cette étape existe maintenant
Si tu refactorises sans structure cible explicite, tu vas simplement déplacer du code au hasard. La structure miroir de `tests/` est intégrée ici pour soutenir immédiatement les couches que tu vas introduire.

État de départ exact
Le frontend fonctionne encore, mais son organisation peut être trop plate ou mélanger plusieurs responsabilités.

Étapes

  1. Dans le terminal Ubuntu déjà ouvert, reste dans `~/workspace/katasensei/katasensei-front`. Vérifie ensuite ton dossier courant : 
    pwd
  2. Crée l'arborescence cible côté source. Copie exactement ces commandes : 
    mkdir -p src/domain/models src/domain/types src/domain/ports src/domain/usecases
mkdir -p src/infrastructure/api/dto src/infrastructure/api/mappers
mkdir -p src/stores
mkdir -p src/ui/composables src/ui/views
    Puis vérifie le résultat : 
    find src -maxdepth 3 -type d | sort
    Résultat attendu : tu vois apparaître `src/domain`, `src/infrastructure`, `src/stores` et `src/ui` avec leurs sous-dossiers utiles à cette phase.
  3. Crée ensuite la structure miroir minimale des tests. Copie : 
    mkdir -p tests/domain/types tests/domain/usecases
mkdir -p tests/infrastructure/api/mappers tests/infrastructure/api
mkdir -p tests/stores tests/ui/composables
    Puis vérifie : 
    find tests -maxdepth 3 -type d | sort
    Résultat attendu : les dossiers de tests reflètent les dossiers de code refactorisés dans cette phase.
  4. Ouvre ensuite le projet dans l'éditeur et fais un repérage concret des fichiers existants. Pour chaque fichier qui participe aujourd'hui au catalogue de katas, demande-toi s'il relève :
    • du métier pur ;
    • d'un détail HTTP ;
    • de l'état partagé ;
    • ou de l'orchestration UI et du rendu.
    Résultat attendu : tu sais quels fichiers devront être déplacés, réécrits ou remplacés.
  5. Ajoute un mémo simple dans tes notes locales : 
    domain = metier pur
infrastructure = details techniques externes
stores = etat partage pour l'ui
ui = orchestration vue et rendu
    Résultat attendu : ce mémo fixe clairement l'emplacement attendu des fichiers de cette étape.

Erreurs fréquentes

  • Ne crée pas `src/ui/components` dans cette phase. Garde le périmètre centré sur le flux de données et la structure front.
  • Ne range pas un DTO d'API dans `src/domain/`. Un DTO décrit un contrat externe, pas une vérité métier.
  • Ne crée pas une structure miroir gigantesque si tu n'as encore aucun test correspondant. Garde-la utile et proportionnée à la phase.

Exercice

Dans quel dossier du projet placerais-tu les éléments suivants : KataRepository, ApiKataDto, GetKatasUseCase, useKataList ?

Solution - A consulter après 20 min
  • KataRepository : `src/domain/ports`
  • ApiKataDto : `src/infrastructure/api/dto`
  • GetKatasUseCase : `src/domain/usecases`
  • useKataList : `src/ui/composables`

Questions théoriques

1. Pourquoi la structure miroir de `tests/` aide-t-elle un refactoring ?

Réponse attendue

Parce qu'elle rend immédiatement visible ce qui est testé et ce qui ne l'est pas. Elle limite aussi le risque de tests fourre-tout qui mélangent plusieurs couches.

2. Pourquoi `useKataList` n'appartient-il pas au domaine ?

Réponse attendue

Parce qu'il orchestre un cas d'usage pour l'UI Vue. Il dépend du cycle de vie ou de l'usage d'un store, donc il ne relève pas du métier pur.

3. Pourquoi faut-il éviter de créer tous les dossiers imaginables d'un coup ?

Réponse attendue

Parce qu'une architecture utile doit rester concrète. Une arborescence trop large au départ fabrique du vide et ajoute de la confusion au lieu de clarifier.

Checklist manuelle

  • Les dossiers `src/domain`, `src/infrastructure`, `src/stores` et `src/ui` existent.
  • Les dossiers de tests miroir utiles existent.
  • Je sais expliquer le rôle de chaque couche.
Commit recommandé : chore(front): pose structure front et miroir de tests

3

Créer les types métier et les erreurs applicatives

Objectif
Introduire les types minimums qui sécurisent le flux du catalogue avant de brancher repository, use case et store.

Pourquoi cette étape existe maintenant
Si tu repousses les types métier après le repository, tu laisses les détails techniques contaminer trop tôt le reste du front.

État de départ exact
Les dossiers sont prêts, mais le front manipule encore des `string` et des objets API trop larges.

Étapes

  1. Crée le modèle métier `Kata` dans `src/domain/models/Kata.ts`. Ouvre ce fichier dans ton éditeur et colle d'abord ce contenu minimal : 
    export type NiveauKata = 'debutant' | 'intermediaire' | 'avance'

export interface Kata {
  id: KataId
  titre: string
  consigne: string
  niveau: NiveauKata
}
    Si ton éditeur signale que `KataId` est inconnu, c'est normal : tu vas le créer juste après.
  2. Crée ensuite `src/domain/types/branded.ts`. Commence par un seul identifiant brandé utile à cette phase : 
    export type KataId = string & { readonly _brand: 'KataId' }

export const toKataId = (value: string): KataId => value as KataId
    Retourne ensuite dans `src/domain/models/Kata.ts` et remplace le contenu précédent par cette version complète avec import : 
    import type { KataId } from '@/domain/types/branded'

export type NiveauKata = 'debutant' | 'intermediaire' | 'avance'

export interface Kata {
  id: KataId
  titre: string
  consigne: string
  niveau: NiveauKata
}
    Résultat attendu : seul ce fichier contient le `as` nécessaire au branding et `Kata.ts` compile sans import manquant.
  3. Crée `src/domain/types/AsyncState.ts`. Colle ce squelette minimal : 
    export type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string }
    Résultat attendu : l'état de chargement devient un contrat explicite au lieu d'être dispersé.
  4. Crée `src/domain/types/Result.ts`. Colle d'abord la base suivante : 
    export type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E }

export const ok = <T>(value: T): Result<T, never> => ({ ok: true, value })
export const err = <E>(error: E): Result<never, E> => ({ ok: false, error })
  5. Crée `src/domain/types/AppError.ts`. Commence par la structure suivante : 
    export type AppError =
  | { kind: 'network'; message: string }
  | { kind: 'validation'; message: string }
  | { kind: 'unexpected'; message: string }
    N'écris pas encore `toUserMessage` au hasard : fais-le proprement dans l'exercice ci-dessous.
  6. Ajoute les trois fichiers de test ciblés dans `tests/domain/types/`. Commence par `tests/domain/types/branded.test.ts` : 
    import { describe, expect, it } from 'vitest'
import { toKataId } from '@/domain/types/branded'

describe('branded', () => {
  it('convertit une string en KataId', () => {
    const kataId = toKataId('kata-1')

    expect(kataId).toBe('kata-1')
  })
})
    Crée ensuite `tests/domain/types/Result.test.ts` : 
    import { describe, expect, it } from 'vitest'
import { err, ok } from '@/domain/types/Result'

describe('Result', () => {
  it('cree un succes avec ok()', () => {
    expect(ok(['kata-1'])).toEqual({
      ok: true,
      value: ['kata-1']
    })
  })

  it('cree un echec avec err()', () => {
    expect(err({ kind: 'network', message: 'offline' })).toEqual({
      ok: false,
      error: { kind: 'network', message: 'offline' }
    })
  })
})
    Termine par `tests/domain/types/AppError.test.ts` : 
    import { describe, expect, it } from 'vitest'
import { toUserMessage } from '@/domain/types/AppError'

describe('toUserMessage', () => {
  it('retourne un message pour network', () => {
    expect(toUserMessage({ kind: 'network', message: 'HTTP 500' }))
      .toBe("Le serveur KataSensei est inaccessible pour le moment.")
  })

  it('retourne un message pour validation', () => {
    expect(toUserMessage({ kind: 'validation', message: 'payload invalide' }))
      .toBe('Les donnees recues sont invalides.')
  })

  it('retourne un message pour unexpected', () => {
    expect(toUserMessage({ kind: 'unexpected', message: 'boom' }))
      .toBe('Une erreur inattendue est survenue.')
  })
})
    Résultat attendu : tu as un test dédié par fichier de type, avec tous les imports nécessaires déjà en place.
  7. Lance ensuite les vérifications de cette étape : 
    npm run type-check
npm run test -- tests/domain/types
    Résultat attendu : le typage passe et les tests ciblés passent.

Erreurs fréquentes

  • Ne fais pas du branding sur tout et n'importe quoi. Commence par les identifiants où l'erreur de confusion est probable.
  • Ne mets pas un message utilisateur brut dans le type d'erreur si tu veux encore pouvoir distinguer la nature de l'erreur.
  • Ne garde pas un modèle `Kata` identique au DTO si tu veux vraiment protéger le domaine des variations HTTP.

Exercice

Dans `src/domain/types/AppError.ts`, écris maintenant la fonction toUserMessage(error: AppError): string. Elle doit :

  • recevoir un objet de type AppError ;
  • utiliser switch (error.kind) ;
  • retourner un message spécifique pour network, validation et unexpected ;
  • ajouter un bloc default avec une variable typée never pour forcer l'exhaustivité si un nouveau cas apparaît ensuite.
Solution - A consulter après 20 min 
export type AppError =
  | { kind: 'network'; message: string }
  | { kind: 'validation'; message: string }
  | { kind: 'unexpected'; message: string }

export function toUserMessage(error: AppError): string {
  switch (error.kind) {
    case 'network':
      return "Le serveur KataSensei est inaccessible pour le moment."
    case 'validation':
      return "Les donnees recues sont invalides."
    case 'unexpected':
      return "Une erreur inattendue est survenue."
    default: {
      const exhaustive: never = error
      return exhaustive
    }
  }
}

Questions théoriques

1. Pourquoi convertir un identifiant API en `KataId` dès l'entrée dans le front ?

Réponse attendue

Parce qu'on veut travailler le plus tôt possible avec un type métier sûr, au lieu de laisser une simple `string` circuler partout.

2. Pourquoi un `Result` explicite est-il souvent plus lisible qu'un `throw` diffus dans le flux front ?

Réponse attendue

Parce que chaque appelant voit directement dans le type qu'il doit gérer deux branches : succès ou erreur. Le contrat devient visible et testable.

3. Pourquoi `AsyncState` est-il préférable à trois variables comme `loading`, `data` et `error` quand le flux devient plus riche ?

Réponse attendue

Parce qu'une union discriminée encode les états autorisés et réduit les combinaisons incohérentes, par exemple `loading=true` avec une erreur déjà affichée.

Checklist manuelle

  • `KataId` existe et son helper est centralisé.
  • `AsyncState` et `Result` existent.
  • `AppError` et `toUserMessage` existent.
  • `npm run type-check` passe après cette étape.
Commit recommandé : feat(front): ajout types metier et erreurs applicatives

4

Isoler le contrat API avec DTO et mapper

Objectif
Empêcher le format HTTP brut de contaminer le domaine et l'UI.

Pourquoi cette étape existe maintenant
Le mapper est le point charnière entre le monde extérieur et le langage métier. Il doit exister avant le repository concret.

État de départ exact
Les types métier du domaine existent, mais la forme de la réponse HTTP n'est pas encore isolée.

Étapes

  1. Crée `src/infrastructure/api/dto/ApiKataDto.ts` avec la forme exacte renvoyée par l'API backend de phase 1. Si tu hésites sur les noms de champs, relis la réponse réelle du backend ou le contrat déjà présent en phase 1. Ne devine pas. Colle d'abord ce fichier complet : 
    export interface ApiKataDto {
  id: string
  titre: string
  consigne: string
  niveau: 'FACILE' | 'MOYEN' | 'DIFFICILE'
}
    Le backend (phase 1) envoie le champ consigne et les niveaux en majuscules : FACILE, MOYEN, DIFFICILE. Le DTO doit refléter exactement ce contrat HTTP. La conversion vers les valeurs du domaine front (debutant, intermediaire, avance) se fait dans le mapper.
  2. Crée `src/infrastructure/api/mappers/KataMapper.ts`. Ce fichier ne doit pas redéclarer `ApiKataDto`. Colle directement ce fichier complet, imports compris : 
    import type { Kata, NiveauKata } from '@/domain/models/Kata'
import { toKataId } from '@/domain/types/branded'
import type { ApiKataDto } from '@/infrastructure/api/dto/ApiKataDto'

const niveauMap: Record<ApiKataDto['niveau'], NiveauKata> = {
  FACILE: 'debutant',
  MOYEN: 'intermediaire',
  DIFFICILE: 'avance'
}

export const KataMapper = {
  toDomain(dto: ApiKataDto): Kata {
    return {
      id: toKataId(dto.id),
      titre: dto.titre,
      consigne: dto.consigne,
      niveau: niveauMap[dto.niveau]
    }
  },

  toDto(kata: Kata): ApiKataDto {
    const niveauBackMap: Record<NiveauKata, ApiKataDto['niveau']> = {
      debutant: 'FACILE',
      intermediaire: 'MOYEN',
      avance: 'DIFFICILE'
    }
    return {
      id: kata.id,
      titre: kata.titre,
      consigne: kata.consigne,
      niveau: niveauBackMap[kata.niveau]
    }
  }
}
    Résultat attendu : le mapper importe bien `ApiKataDto`, expose `toDomain(dto)` et `toDto(kata)`, et ne redéclare aucun type API dans le mauvais dossier.
  3. Écris les tests du mapper dans `tests/infrastructure/api/mappers/KataMapper.test.ts`. Commence par un fichier réellement copiable : 
    import { describe, expect, it } from 'vitest'
import { KataMapper } from '@/infrastructure/api/mappers/KataMapper'
import type { ApiKataDto } from '@/infrastructure/api/dto/ApiKataDto'

describe('KataMapper', () => {
  it('convertit un dto api en kata de domaine', () => {
    const dto: ApiKataDto = {
      id: 'kata-1',
      titre: 'FizzBuzz',
      consigne: 'Ecris FizzBuzz',
      niveau: 'FACILE'
    }

    const kata = KataMapper.toDomain(dto)

import type { AppError } from '@/domain/types/AppError'
import type { Result } from '@/domain/types/Result'

import type { KataRepository } from '@/domain/ports/KataRepository'
import type { ApiKataDto } from '@/infrastructure/api/dto/ApiKataDto'

export class ApiKataRepository implements KataRepository {
  async findAll(): Promise<Result<Kata[], AppError>> {
    try {
      const response = await fetch('/api/katas')

      if (!response.ok) {
        return err({ kind: 'network', message: `HTTP ${response.status}` })
      }

      const data: ApiKataDto[] = await response.json()
      return ok(data.map(KataMapper.toDomain))
    } catch {
      return err({ kind: 'network', message: 'Serveur inaccessible' })
    }

    try {
      const response = await fetch(`/api/katas/${id}`)

      if (!response.ok) {
        return err({ kind: 'network', message: `HTTP ${response.status}` })
      }

      const data: ApiKataDto = await response.json()
      return ok(KataMapper.toDomain(data))
    } catch {
      return err({ kind: 'network', message: 'Serveur inaccessible' })
    }
  }
}

Erreurs fréquentes

  • Un type-check vert ne suffit pas si la vue ne fonctionne plus. Vérifie aussi le comportement visible.
  • Des tests trop couplés à l'implémentation peuvent casser au moindre déplacement de fichier. Vérifie qu'ils testent bien le comportement utile.
  • Ne remets pas en dernière minute des raccourcis techniques dans la vue pour faire passer une vérification rapide.

Questions théoriques

1. Pourquoi une phase de refactoring doit-elle se finir par une preuve utilisateur visible ?

Réponse attendue

Parce qu'un refactoring sert le produit. Si la fonctionnalité n'est plus observable ou n'est plus fiable, la dette technique n'a pas vraiment diminué.

2. Pourquoi vérifier aussi l'arborescence et pas seulement les tests ?

Réponse attendue

Parce qu'une architecture est aussi un moyen de rendre les responsabilités visibles. Les tests valident le comportement, l'arborescence aide à garder la lisibilité du système.

3. Pourquoi la phase 3 peut-elle maintenant se concentrer sur `PrimeVue` ?

Réponse attendue

Parce que le flux de données et les points d'extension du front sont désormais plus stables. Les composants UI métier basés sur `PrimeVue` viendront s'appuyer sur une base moins fragile.

Stratégie de déploiement de fin de phase

Cette phase 2 doit se terminer par un redéploiement. Le but n'est pas de changer la plateforme de livraison, mais de prouver qu'un refactoring architectural frontend n'a pas cassé la capacité à mettre KataSensei en ligne.

  • Avant déploiement : exécute `npm run type-check`, puis `npm run build`, puis `npm run test -- tests/domain/types tests/domain/usecases tests/infrastructure/api tests/stores tests/ui/composables`.
  • Avant déploiement : relance ensuite le frontend avec `npm run dev` et vérifie dans le navigateur que le catalogue s'affiche encore correctement en local.

  • Après déploiement : vérifie qu'un backend indisponible déclenche encore un message d'erreur simple au lieu d'un plantage silencieux.
  • La preuve attendue en fin de phase est la suivante : le même incrément utilisateur qu'en phase 1 reste bien livrable après refactoring architectural.

Ce que tu sais faire après cette phase

  • Refactoriser un flux front visible sans casser la fonctionnalité utilisateur.
  • Créer des types métier utiles côté frontend sans sur-abstraire.
  • Isoler un contrat HTTP derrière un DTO et un mapper.
  • Faire dépendre un use case front d'un port plutôt que d'un adapter concret.
  • Utiliser Pinia comme source de vérité UI au lieu d'y faire du réseau.
  • Structurer les tests frontend en miroir du code refactorisé.
À retenir : la phase 2 ne cherche pas à fabriquer encore toute l'interface finale. Elle prépare un terrain stable pour que la phase 3 puisse construire de vrais composants métier avec `PrimeVue` sans empiler la dette technique.

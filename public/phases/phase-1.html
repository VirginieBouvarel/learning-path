Phase 1

Premier catalogue de katas persisté, testé et visible côté front

Durée estimée : ~10h à ~14h réparties sur 2 semaines

La phase 0 t'a permis de construire un socle fullstack minimal : le backend répond, le frontend s'affiche, et les deux savent déjà communiquer. La phase 1 transforme ce socle en premier vrai incrément métier. À la fin, KataSensei saura lire et créer des katas dans PostgreSQL, renvoyer des réponses HTTP cohérentes, puis afficher le catalogue côté Vue.

⚫Backend ⚫Persistance ⚫Frontend

Objectif de phase

Construire un premier catalogue de katas persisté en base, accessible via une API REST simple, puis affiché dans le frontend Vue.

État de départ exact

  • Tu travailles sur Windows 11 mais toutes les commandes du projet se font dans Ubuntu via WSL2.
  • Le repo KataSensei existe déjà dans WSL.
  • Le backend Spring Boot démarre localement.
  • Le frontend Vue démarre localement.
  • Le frontend sait déjà appeler au moins un endpoint simple du backend hérité de la phase 0.
  • Le projet n'a pas encore de persistance métier réelle pour les katas.

Résultat attendu en fin de phase

  • PostgreSQL démarre localement pour KataSensei.
  • Le backend lit sa configuration de base via variables d'environnement.
  • Une migration Flyway crée la table `kata` et ajoute des données de départ.
  • `GET /katas` renvoie la liste des katas.
  • `POST /katas` crée un nouveau kata avec validation.
  • `GET /katas/{id}` renvoie un kata existant ou une erreur JSON homogène.
  • Le frontend affiche la liste réelle des katas.
  • Le projet possède un premier minimum de commandes de qualité avant commit.

Valeur livrée à l'utilisateur

KataSensei cesse d'être un simple squelette technique et devient un premier catalogue consultable, avec une vraie donnée persistée.

Prérequis

  • La phase 0 est terminée et les deux applications redémarrent encore.
  • Tu sais ouvrir plusieurs terminaux Ubuntu dans WSL.
  • Docker Desktop côté Windows, ou une installation Docker équivalente, est fonctionnelle avec WSL.

Outils mobilisés

  • Terminal Ubuntu dans WSL
  • Docker Compose
  • PostgreSQL
  • Spring Boot
  • Flyway
  • JUnit 5, MockMvc et AssertJ
  • Vue 3, TypeScript strict et Vitest
  • ESLint, Prettier et Husky
1

Repartir proprement de la phase 0 et vérifier le point de départ

Objectif
Vérifier que le socle de phase 0 fonctionne encore avant d'ajouter de la persistance et du code métier.

Pourquoi cette étape existe maintenant
Si le front ou le back est déjà cassé avant de commencer, tu ne pourras plus distinguer un problème ancien d'un problème introduit par la phase 1.

État de départ exact
Le repo existe déjà, mais rien ne garantit que les applications sont encore lancées ou que leur démarrage fonctionne encore aujourd'hui.

Étapes

  1. Ouvre un terminal Ubuntu dans WSL.
  2. Place-toi à la racine du repo : 
    cd ~/workspace/katasensei
    Résultat attendu : `pwd` affiche le dossier du projet.
  3. Vérifie que le repo contient bien un backend et un frontend : 
    ls
ls katasensei-back
ls katasensei-front
    Résultat attendu : tu vois les deux dossiers applicatifs.
  4. Dans un premier terminal, démarre le backend : 
    cd ~/workspace/katasensei/katasensei-back
./mvnw spring-boot:run
    Résultat attendu : Spring Boot démarre sans erreur fatale et écoute sur `http://localhost:8080`.
  5. Dans un second terminal, démarre le frontend : 
    cd ~/workspace/katasensei/katasensei-front
npm run dev
    Résultat attendu : Vite écoute sur `http://localhost:5173`.
  6. Dans un troisième terminal, vérifie les routes minimales héritées de la phase 0 : 
    curl http://localhost:8080/actuator/health
curl http://localhost:8080/api/ping
    Résultat attendu : la première commande retourne un JSON avec `UP`, la seconde un JSON métier simple.
  7. Ouvre `http://localhost:5173` dans ton navigateur. Résultat attendu : le frontend s'affiche encore sans erreur bloquante.

Erreurs fréquentes

  • Si `./mvnw` échoue avec une erreur de permission, lance `chmod +x mvnw` dans `katasensei-back`.
  • Si `npm run dev` échoue, vérifie d'abord que tu te trouves bien dans `katasensei-front`.
  • Si `curl` ne répond pas, attends quelques secondes de plus ou relis les logs de démarrage.

Exercice

Écris en une phrase ce que la phase 1 doit livrer côté produit, sans parler de framework ni d'outil.

Solution - A consulter après 20 min

Une réponse correcte est : "l'application doit permettre de consulter un premier catalogue de katas persisté et affiché dans l'interface".

Questions théoriques

1. Pourquoi vérifier le socle de phase 0 avant d'ajouter du nouveau code ?

Réponse attendue

Parce que sinon tu risques de diagnostiquer au mauvais endroit. Il faut partir d'un état stable pour comprendre ce qui casse réellement ensuite.

2. Pourquoi distinguer trois terminaux différents pendant le développement local ?

Réponse attendue

Parce que le backend, le frontend et les commandes de vérification n'ont pas le même rôle. Les séparer rend les logs et les actions plus lisibles.

3. Pourquoi la valeur métier de la phase doit-elle être formulée sans jargon technique ?

Réponse attendue

Parce qu'une phase sert d'abord à construire une fonctionnalité utile. Les outils sont des moyens, pas le livrable final.

Definition of Done de l'étape

  • Le frontend et le backend redémarrent encore.
  • Les routes héritées de la phase 0 répondent.
  • Tu sais décrire en une phrase le livrable métier de la phase 1.

Checklist manuelle

  • J'ai relancé le backend.
  • J'ai relancé le frontend.
  • `GET /actuator/health` répond toujours.
  • Je sais expliquer le but métier de la phase 1.
2

Démarrer PostgreSQL localement et brancher les variables d'environnement

Objectif
Préparer une base locale reproductible pour stocker les premiers katas.

Pourquoi cette étape existe maintenant
La phase 0 n'avait pas encore besoin d'une base. La phase 1 en a besoin dès qu'on veut conserver de vraies données entre deux redémarrages.

État de départ exact
Le backend fonctionne encore sans PostgreSQL métier. Le projet n'est pas encore connecté à une vraie base locale.

Avant de commencer : vocabulaire minimum

Docker Compose permet de démarrer facilement un service externe, ici PostgreSQL, sans installation manuelle longue sur la machine.

Variable d'environnement = une valeur donnée à l'application au moment du démarrage, comme un nom de base ou un mot de passe, sans l'écrire en dur dans le code Java.

Étapes

  1. Ouvre un terminal Ubuntu et place-toi à la racine du repo : 
    cd ~/workspace/katasensei
  2. Crée le fichier `docker-compose.yml` à la racine du repo : 
    touch docker-compose.yml
  3. Écris le contenu minimal suivant dans `docker-compose.yml` : 
    services:
  postgres:
    image: postgres:16
    container_name: katasensei-postgres
    environment:
      POSTGRES_DB: katasensei
      POSTGRES_USER: katasensei
      POSTGRES_PASSWORD: katasensei
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
  4. Démarre PostgreSQL : 
    docker compose up -d postgres
    Résultat attendu : le conteneur démarre en arrière-plan.
  5. Vérifie que PostgreSQL tourne : 
    docker compose ps
    Résultat attendu : tu vois un service `postgres` en état `running` ou `up`.
  6. Teste l'accès à la base : 
    docker exec -it katasensei-postgres psql -U katasensei -d katasensei -c '\l'
    Résultat attendu : tu vois la liste des bases, dont `katasensei`.
  7. Dans le backend, ouvre le fichier de configuration Spring, généralement `src/main/resources/application.yml` ou `src/main/resources/application.properties`.
  8. Ajoute ou adapte la configuration datasource pour lire les variables d'environnement. Exemple avec `application.yml` : 
    spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST}:${DB_PORT}/${DB_NAME}
    username: ${DB_USER}
    password: ${DB_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate
  flyway:
    enabled: true
  9. Dans le terminal qui servira à démarrer le backend, exporte les variables : 
    export DB_HOST=localhost
export DB_PORT=5432
export DB_NAME=katasensei
export DB_USER=katasensei
export DB_PASSWORD=katasensei
  10. Relance le backend : 
    cd ~/workspace/katasensei/katasensei-back
./mvnw spring-boot:run

Erreurs fréquentes

  • Si `docker compose` ne fonctionne pas, vérifie d'abord que Docker est bien lancé côté Windows.
  • Si Spring échoue à se connecter à la base, vérifie l'ordre suivant : variables exportées, port 5432, nom de base, utilisateur, mot de passe.
  • Si tu utilises `application.properties` au lieu de `application.yml`, adapte simplement la syntaxe, mais garde les mêmes valeurs.

Exercice

Explique avec tes mots pourquoi on utilise des variables d'environnement au lieu d'écrire l'URL complète et le mot de passe directement en dur dans le code Java.

Solution - A consulter après 20 min

Parce qu'on veut pouvoir changer l'environnement de connexion sans modifier le code source, et parce qu'on évite de stocker des informations sensibles directement dans les classes Java.

Questions théoriques

1. Pourquoi PostgreSQL arrive-t-il en phase 1 et pas en phase 0 ?

Réponse attendue

Parce que la phase 0 devait seulement prouver que le système fullstack démarrait. La persistance devient utile seulement quand un vrai besoin métier apparaît.

2. Pourquoi `docker compose` est-il utile pour une base locale de développement ?

Réponse attendue

Parce qu'il permet de redémarrer facilement le même service sur n'importe quelle machine du projet sans refaire toute une installation manuelle.

3. Pourquoi `ddl-auto: validate` est-il plus cohérent ici que `create` ?

Réponse attendue

Parce qu'on veut que le schéma soit piloté par Flyway, pas reconstruit silencieusement par Hibernate à chaque démarrage.

Definition of Done de l'étape

  • PostgreSQL démarre localement via Docker Compose.
  • Le backend connaît ses paramètres de connexion via variables d'environnement.
  • Tu sais vérifier que la base répond réellement.

Checklist manuelle

  • `docker compose ps` montre PostgreSQL démarré.
  • Je peux me connecter à la base avec `psql` dans le conteneur.
  • Le backend lit sa datasource depuis des variables.
Commit recommandé : feat(infra): ajout postgres local et configuration datasource
3

Poser le schéma initial avec Flyway

Objectif
Créer la première version du schéma de base de données pour stocker les katas.

Pourquoi cette étape existe maintenant
Une fois PostgreSQL disponible, il faut définir explicitement la structure de la table métier avant d'écrire l'API qui la consomme.

État de départ exact
Le backend sait se connecter à PostgreSQL, mais la base ne contient encore aucune table métier `kata`.

Avant de commencer : vocabulaire minimum

Flyway est un outil de migration de base de données. Une migration est simplement un fichier versionné qui décrit un changement du schéma SQL.

Exemple : créer une table, ajouter une colonne, insérer des données de départ.

Étapes

  1. Ouvre un terminal Ubuntu et place-toi dans le backend : 
    cd ~/workspace/katasensei/katasensei-back
  2. Crée le dossier des migrations s'il n'existe pas : 
    mkdir -p src/main/resources/db/migration
  3. Crée le fichier `src/main/resources/db/migration/V1__create_kata_table.sql`.
  4. Écris le contenu suivant : 
    CREATE TABLE kata (
    id UUID PRIMARY KEY,
    titre VARCHAR(255) NOT NULL,
    consigne TEXT NOT NULL,
    niveau VARCHAR(20) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);

INSERT INTO kata (id, titre, consigne, niveau) VALUES
    ('00000000-0000-0000-0000-000000000001', 'FizzBuzz', 'Retourne Fizz, Buzz ou FizzBuzz selon le nombre', 'FACILE'),
    ('00000000-0000-0000-0000-000000000002', 'Palindrome', 'Detecte si un texte se lit dans les deux sens', 'FACILE'),
    ('00000000-0000-0000-0000-000000000003', 'Anagramme', 'Detecte si deux mots ont les memes lettres', 'MOYEN');
  5. Relance le backend avec les variables d'environnement déjà exportées. Résultat attendu : Flyway exécute la migration au démarrage.
  6. Vérifie que la table existe : 
    docker exec -it katasensei-postgres psql -U katasensei -d katasensei -c '\dt'
    Résultat attendu : tu vois la table `kata`.
  7. Vérifie les données insérées : 
    docker exec -it katasensei-postgres psql -U katasensei -d katasensei -c 'SELECT id, titre, niveau FROM kata;'
    Résultat attendu : tu vois les trois lignes de seed.

Erreurs fréquentes

  • Si la table n'apparaît pas, relis d'abord les logs Spring pour voir si Flyway a vraiment tourné.
  • Si tu modifies `V1` après exécution, Flyway signale une incohérence. En local, repars d'une base propre puis relance la migration au lieu d'éditer une migration déjà jouée.
  • Si `niveau` accepte n'importe quelle valeur, tu repousses un problème de cohérence. Garde dès maintenant un ensemble simple de valeurs métier.

Exercice

Ajoute un quatrième kata de niveau `DIFFICILE`, mais sans modifier `V1` une fois la migration déjà exécutée.

Solution - A consulter après 20 min

La bonne solution est de créer une nouvelle migration, par exemple `V2__insert_hard_kata.sql`, plutôt que de réécrire une migration déjà jouée.

Questions théoriques

1. Pourquoi versionner le schéma SQL au lieu de créer la table à la main dans un outil graphique ?

Réponse attendue

Parce qu'un fichier versionné est traçable, relisible et reproductible. Toute l'équipe peut recréer le même schéma.

2. Pourquoi insérer des seeds dès cette phase ?

Réponse attendue

Parce qu'elles donnent une preuve immédiate que la lecture depuis la base fonctionne réellement, sans devoir tout créer manuellement avant chaque test.

3. Pourquoi vérifier la base directement avec `psql` après la migration ?

Réponse attendue

Parce que les logs de démarrage ne suffisent pas. La base elle-même doit confirmer que la table et les lignes existent vraiment.

Definition of Done de l'étape

  • Une migration Flyway versionnée existe.
  • La table `kata` existe dans PostgreSQL.
  • Des données de départ sont présentes et visibles.

Checklist manuelle

  • Le fichier `V1__create_kata_table.sql` existe au bon endroit.
  • La table `kata` existe dans PostgreSQL.
  • Les trois katas seedés sont présents.
Commit recommandé : feat(database): ajout migration initiale des katas
4

Lire les katas depuis le backend avec un premier test guidé

Objectif
Créer le premier endpoint métier réel du backend : GET /katas.

Pourquoi cette étape existe maintenant
La base PostgreSQL et la migration existent déjà. On a donc enfin quelque chose de concret à lire. C'est le bon moment pour apprendre un premier test backend, un controller, un service, un repository et un DTO de réponse.

État de départ exact
PostgreSQL tourne, la table `kata` existe et contient des lignes de seed, mais le backend n'expose pas encore `GET /katas`.

Avant de commencer : vocabulaire minimum

Test backend = fichier Java qui vérifie automatiquement un comportement du serveur.

Test web = test centré sur la couche HTTP, ici l'URL, le code de retour et le JSON.

Controller = classe qui reçoit la requête HTTP.

Service = classe qui porte une logique applicative simple.

Repository = interface qui lit et écrit dans la base via Spring Data JPA.

DTO de réponse = objet Java renvoyé au frontend, pensé pour l'API.

Étapes

  1. Dans `katasensei-back`, ouvre `pom.xml` et vérifie la présence de la dépendance de test : 
    <dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-test</artifactId>
  <scope>test</scope>
</dependency>
    Résultat attendu : cette dépendance est présente. Elle apporte JUnit 5, MockMvc et AssertJ.
  2. Vérifie ou crée le dossier de test du package `kata`. Si ton package racine est `dev.katasensei`, crée : 
    cd ~/workspace/katasensei/katasensei-back
mkdir -p src/test/java/dev/katasensei/kata
  3. Crée le fichier `src/test/java/dev/katasensei/kata/KataControllerWebTest.java`.
  4. Écris d'abord ce test web : 
    package dev.katasensei.kata;

import static org.mockito.BDDMockito.given;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.jsonPath;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;

import java.util.List;
import java.util.UUID;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.WebMvcTest;
import org.springframework.boot.test.mock.mockito.MockBean;
import org.springframework.test.web.servlet.MockMvc;

@WebMvcTest(KataController.class)
class KataControllerWebTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private KataService kataService;

    @Test
    void should_return_kata_list_when_endpoint_is_called() throws Exception {
        given(kataService.lister()).willReturn(List.of(
            new KataResponse(
                UUID.fromString("00000000-0000-0000-0000-000000000001"),
                "FizzBuzz",
                "FACILE"
            )
        ));

        mockMvc.perform(get("/katas"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$[0].titre").value("FizzBuzz"))
            .andExpect(jsonPath("$[0].niveau").value("FACILE"));
    }
}
  5. Lance les tests : 
    ./mvnw test
    Résultat attendu : le test échoue parce que `KataController`, `KataService` et `KataResponse` n'existent pas encore.
  6. Crée le fichier `src/main/java/dev/katasensei/kata/KataResponse.java` : 
    package dev.katasensei.kata;

import java.util.UUID;

public record KataResponse(
    UUID id,
    String titre,
    String niveau
) {
}
  7. Crée le fichier `src/main/java/dev/katasensei/kata/KataService.java` : 
    package dev.katasensei.kata;

import java.util.List;

import org.springframework.stereotype.Service;

@Service
public class KataService {

    public List<KataResponse> lister() {
        return List.of();
    }
}
  8. Crée le fichier `src/main/java/dev/katasensei/kata/KataController.java` : 
    package dev.katasensei.kata;

import java.util.List;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class KataController {

    private final KataService kataService;

    public KataController(KataService kataService) {
        this.kataService = kataService;
    }

    @GetMapping("/katas")
    public List<KataResponse> lister() {
        return kataService.lister();
    }
}
  9. Relance les tests : 
    ./mvnw test
    Résultat attendu : le test web passe maintenant.
  10. Crée l'entité `src/main/java/dev/katasensei/kata/Kata.java` : 
    package dev.katasensei.kata;

import java.time.LocalDateTime;
import java.util.UUID;

import jakarta.persistence.Column;
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Table;

@Entity
@Table(name = "kata")
public class Kata {

    @Id
    private UUID id;

    @Column(nullable = false)
    private String titre;

    @Column(nullable = false)
    private String consigne;

    @Column(nullable = false)
    private String niveau;

    @Column(name = "created_at", nullable = false)
    private LocalDateTime createdAt;

    protected Kata() {
    }

    public UUID getId() {
        return id;
    }

    public String getTitre() {
        return titre;
    }

    public String getConsigne() {
        return consigne;
    }

    public String getNiveau() {
        return niveau;
    }

    public LocalDateTime getCreatedAt() {
        return createdAt;
    }
}
  11. Crée le repository `src/main/java/dev/katasensei/kata/KataRepository.java` : 
    package dev.katasensei.kata;

import java.util.UUID;

import org.springframework.data.jpa.repository.JpaRepository;

public interface KataRepository extends JpaRepository<Kata, UUID> {
}
  12. Remplace le contenu de `KataService.java` par une vraie lecture depuis la base : 
    package dev.katasensei.kata;

import java.util.List;

import org.springframework.stereotype.Service;

@Service
public class KataService {

    private final KataRepository kataRepository;

    public KataService(KataRepository kataRepository) {
        this.kataRepository = kataRepository;
    }

    public List<KataResponse> lister() {
        return kataRepository.findAll()
            .stream()
            .map(this::toResponse)
            .toList();
    }

    private KataResponse toResponse(Kata kata) {
        return new KataResponse(
            kata.getId(),
            kata.getTitre(),
            kata.getNiveau()
        );
    }
}
  13. Crée le test d'intégration `src/test/java/dev/katasensei/kata/KataServiceIntegrationTest.java` : 
    package dev.katasensei.kata;

import static org.assertj.core.api.Assertions.assertThat;

import java.util.List;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;

@SpringBootTest
class KataServiceIntegrationTest {

    @Autowired
    private KataService kataService;

    @Test
    void should_read_seeded_katas_from_database() {
        List<KataResponse> result = kataService.lister();

        assertThat(result).isNotEmpty();
        assertThat(result)
            .extracting(KataResponse::titre)
            .contains("FizzBuzz");
    }
}
  14. Lance à nouveau les tests : 
    ./mvnw test
    Résultat attendu : le test web passe, le test d'intégration passe, et Maven termine avec `BUILD SUCCESS`.
  15. Vérifie enfin l'endpoint à la main : 
    curl http://localhost:8080/katas
    Résultat attendu : un tableau JSON apparaît avec au moins `FizzBuzz`.

Erreurs fréquentes

  • Si `MockMvc` ou `WebMvcTest` est introuvable, ajoute la dépendance `spring-boot-starter-test` dans `pom.xml` puis relance Maven.
  • Si le test web passe mais que `curl /katas` échoue, le problème se situe sans doute dans la lecture réelle de la base.
  • Si `relation "kata" does not exist`, la migration n'a pas été appliquée à la bonne base.

Exercice

Sans regarder la solution, explique pourquoi on garde deux tests différents dans cette étape au lieu d'un seul gros test.

Solution - A consulter après 20 min

Parce qu'ils ne prouvent pas la même chose. Le test web vérifie la couche HTTP. Le test d'intégration vérifie la vraie lecture depuis la base.

Questions théoriques

1. Quelle est la différence entre `Kata` et `KataResponse` ?

Réponse attendue

Kata est lié à la base via JPA. KataResponse est l'objet renvoyé par l'API HTTP.

2. Pourquoi le controller n'appelle-t-il pas directement PostgreSQL ?

Réponse attendue

Parce que son rôle est de gérer HTTP. La lecture et la transformation des données sont déléguées au service et au repository.

3. Pourquoi le test web utilise-t-il un mock du service ?

Réponse attendue

Parce qu'il veut tester la couche web sans dépendre de la vraie base. Il se concentre sur la route, le statut et le JSON.

Definition of Done de l'étape

  • `KataControllerWebTest` existe et passe.

  • Si tu rends le hook trop lourd, tu seras tentée de le contourner. Garde-le simple en phase 1.
  • Si `npm run test` échoue avec une erreur liée à `jsdom`, vérifie la configuration de `vite.config.ts` et la présence de `jsdom` dans les dépendances.
  • Ne transforme pas cette étape en inventaire d'outils sans rapport avec le flux métier construit dans la phase.

Questions théoriques

1. Pourquoi attendre le premier flux métier réel avant d'insister sur l'outillage qualité ?

Réponse attendue

Parce qu'un outil protège quelque chose. Avant d'avoir un vrai flux métier, il risque surtout d'alourdir le parcours sans bénéfice concret perceptible.

2. Pourquoi un hook Git doit-il rester léger au début ?

Réponse attendue

Parce qu'un garde-fou trop coûteux finit souvent par être désactivé ou contourné.

3. Pourquoi certains sujets comme la CI/CD complète ou Checkstyle détaillé peuvent-ils être reportés à plus tard ?

Réponse attendue

Parce qu'ils seront plus utiles quand l'architecture et les conventions du projet seront plus stables. Ici, l'objectif est d'abord de sécuriser le premier incrément métier.

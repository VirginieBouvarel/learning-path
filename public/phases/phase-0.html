Phase 0

Setup réel du projet — de zéro à un système minimal observable

Durée estimée : ~8h réparties sur 1 semaine

Cette phase part d'une situation réelle : tu es sur Windows 11, tu travailles dans Ubuntu via WSL2, et le projet KataSensei n'existe pas encore sur ta machine. À la fin de la phase, tu auras un repo Git initialisé, un backend Spring Boot qui répond, un frontend Vue qui s'affiche et une première communication HTTP visible entre les deux.

⚫Setup & environnement ⚫Frontend ⚫Backend
WSL2 + Terminal IDE et éditeurs Structure de dossiers

Objectif de phase

Construire une base de travail exécutable sans aucun implicite, depuis un poste Windows 11 + WSL2 Ubuntu, jusqu'à un premier système fullstack minimal.

État de départ exact

  • Windows 11 est installé.
  • WSL2 n'est pas encore forcément installé.
  • Ubuntu n'est pas encore forcément installé dans WSL.
  • zsh et Oh My Zsh ne sont pas encore forcément installés.
  • Le repo KataSensei n'existe pas encore sur ta machine.
  • Tu n'as encore ni projet Spring Boot, ni projet Vue, ni dossier de travail KataSensei.

Résultat attendu en fin de phase

  • Un dossier `~/workspace/katasensei` existe dans WSL.
  • Ce dossier est un repo Git initialisé.
  • Le backend `katasensei-back` démarre sur `http://localhost:8080`.
  • Le health check `http://localhost:8080/actuator/health` répond avec un statut `UP`.
  • Le frontend `katasensei-front` démarre sur `http://localhost:5173`.
  • La page front affiche la réponse du backend.

Valeur livrée à l'utilisateur

La valeur livrée n'est pas encore fonctionnelle côté produit, mais elle est technique et observable : le squelette complet du système tourne, ce qui rend possible la première vraie fonctionnalité métier en phase 1.

Prérequis

  • Tu as un compte Windows 11 avec les droits nécessaires pour installer WSL.
  • Tu peux ouvrir Windows Terminal ou l'invite de commandes Windows pour lancer l'installation de WSL.
  • Tu acceptes de travailler ensuite uniquement avec des commandes Unix dans le terminal Ubuntu.

Outils mobilisés

  • Windows Terminal ou invite de commandes Windows pour l'installation initiale de WSL
  • Terminal Ubuntu dans WSL2
  • zsh et Oh My Zsh
  • Git
  • SDKMAN pour Java et Maven
  • NVM pour Node.js
  • Spring Initializr
  • Vue 3 + Vite + TypeScript
  • Un éditeur de code connecté à WSL
1

Installer WSL2, Ubuntu, zsh et Oh My Zsh

Objectif
Installer l'environnement Unix de travail réel du learning path avant toute commande de projet.

Pourquoi cette étape existe maintenant
Le guide ne peut pas supposer que WSL2, Ubuntu ou `zsh` existent déjà. Il faut d'abord construire le terminal cible du parcours.

État de départ exact
Tu es sur Windows 11. Aucun repo KataSensei n'existe encore. WSL2, Ubuntu, `zsh` et Oh My Zsh peuvent être absents.

Micro-étapes

  1. Ouvre Windows Terminal ou l'invite de commandes Windows en mode administrateur. Cette étape se fait côté Windows, pas encore dans Ubuntu.
  2. Lance l'installation standard de WSL : 
    wsl --install
    Résultat attendu : Windows indique qu'il active WSL, installe Ubuntu et demande un redémarrage.
  3. Redémarre la machine si Windows le demande, puis ouvre Ubuntu depuis le menu Démarrer.
  4. Lors du premier lancement d'Ubuntu, choisis ton nom d'utilisateur Linux et ton mot de passe. Résultat attendu : un terminal Ubuntu s'ouvre avec un prompt de type virginie@machine:~$.
  5. Dans ce terminal Ubuntu, mets à jour les paquets : 
    sudo apt update
sudo apt upgrade -y
  6. Installe `zsh` et Git : 
    sudo apt install -y zsh git curl unzip zip
  7. Vérifie que `zsh` est bien installé : 
    zsh --version
  8. Installe Oh My Zsh : 
    sh -c "$(curl -fsSL https://raw.githubusercontent.com/ohmyzsh/ohmyzsh/master/tools/install.sh)"
    Résultat attendu : le script crée `~/.oh-my-zsh`, configure `~/.zshrc` et peut basculer automatiquement vers `zsh`.
  9. Vérifie quel shell est utilisé : 
    echo $SHELL
    Résultat attendu : tu dois voir un chemin se terminant par /zsh.
  10. Si nécessaire, définis `zsh` comme shell par défaut puis ferme et rouvre Ubuntu : 
    chsh -s $(which zsh)

Résultat attendu

Tu disposes maintenant d'un terminal Ubuntu dans WSL2 avec `zsh` et Oh My Zsh, qui constitue l'environnement officiel du learning path.

Erreurs fréquentes

  • wsl --install échoue : vérifie que tu es bien dans un terminal Windows lancé en administrateur.
  • Ubuntu ne se lance pas après installation : ouvre l'application Ubuntu depuis le menu Démarrer et termine l'initialisation du compte Linux.
  • echo $SHELL n'affiche pas `zsh` : relance Ubuntu ou exécute chsh -s $(which zsh), puis ferme et rouvre le terminal.
  • Le script Oh My Zsh échoue à télécharger : vérifie la connexion réseau puis relance la commande dans Ubuntu.

Exercice

Ferme complètement Ubuntu, rouvre-le, puis vérifie sans aide que tu es bien revenue dans un shell `zsh`.

Solution - A consulter après 20 min 
echo $SHELL
zsh --version

Tu dois voir un chemin finissant par `/zsh` et une version de `zsh` installée.

Questions théoriques

Question 1 Pourquoi installer WSL2 avant même de créer le projet ?

Réponse attendue

Parce que tout le parcours repose sur un environnement Unix cohérent. Sans WSL2, les chemins, les shells, les commandes et certains comportements d'outillage ne correspondent pas au guide.

Question 2 À quoi sert `zsh` dans ce parcours ?

Réponse attendue

`zsh` fournit un shell Unix moderne, confortable et cohérent pour toutes les commandes du learning path. Ce n'est pas obligatoire dans l'absolu pour programmer, mais c'est l'environnement de référence ici.

Question 3 Pourquoi installer Oh My Zsh si ce n'est pas indispensable techniquement ?

Réponse attendue

Parce qu'il améliore l'ergonomie du terminal dès le départ et stabilise l'environnement pédagogique du parcours. On veut éviter les variations inutiles de configuration entre phases.

Definition of Done de l'étape

  • WSL2 est installé.
  • Ubuntu se lance correctement.
  • `zsh` est installé.
  • Oh My Zsh est installé.
  • Le shell actif est `zsh`.

Checklist manuelle

  • J'ai lancé `wsl --install` depuis un terminal Windows adapté.
  • Ubuntu s'ouvre correctement sur ma machine.
  • `zsh --version` répond sans erreur.
  • `echo $SHELL` retourne un chemin finissant par `/zsh`.
  • Oh My Zsh est installé.
2

Préparer l'espace de travail WSL et vérifier les outils minimaux

Objectif
Créer un point de départ propre dans Ubuntu et vérifier que les outils indispensables existent avant de générer les projets.

Pourquoi cette étape existe maintenant
Une fois l'environnement Unix installé, il faut poser un dossier de travail propre et vérifier les outils de base avant de générer les projets.

État de départ exact
Tu es sur Windows 11. WSL2, Ubuntu, `zsh` et Oh My Zsh sont installés. Aucun repo KataSensei n'existe encore. Tu vas tout faire depuis un terminal Ubuntu ouvert dans WSL.

Micro-étapes

  1. Ouvre un terminal Ubuntu dans WSL. Le prompt doit ressembler à quelque chose comme virginie@machine:~$.
  2. Affiche ton dossier courant pour vérifier que tu démarres bien dans ton home Linux : 
    pwd
    Tu dois voir un chemin de type /home/ton-user.
  3. Crée un dossier de travail unique pour tous tes projets si tu ne l'as pas encore : 
    mkdir -p ~/workspace
cd ~/workspace
    Résultat attendu : pwd doit répondre /home/ton-user/workspace.
  4. Vérifie que Git, Curl et Unzip sont disponibles : 
    git --version
curl --version
unzip -v
    Résultat attendu : les trois commandes affichent une version sans erreur.
  5. Si une des commandes précédentes échoue, installe les outils manquants dans le même terminal Ubuntu : 
    sudo apt update
sudo apt install -y git curl unzip zip
    Résultat attendu : l'installation se termine sans message d'erreur bloquant.
  6. Vérifie si Java et Node sont déjà présents : 
    java -version
node -v
npm -v
    Si une commande échoue, ce n'est pas grave. Tu vas les installer dans les étapes suivantes.

Résultat attendu

Tu disposes maintenant d'un terminal Ubuntu prêt à l'emploi, d'un dossier de travail explicite dans WSL et d'une vision claire des outils déjà installés ou manquants.

Erreurs fréquentes

  • pwd retourne un chemin qui commence par /mnt/c/ : tu es dans le système de fichiers Windows. Reviens dans ton home Linux avec cd ~, puis repars dans ~/workspace.
  • sudo: command not found : ton terminal n'est pas Ubuntu. Ferme-le et ouvre explicitement la distribution Ubuntu de WSL.
  • git: command not found : installe Git avec sudo apt install -y git.

Exercice

Ferme le terminal, rouvre un nouveau terminal Ubuntu, puis reviens sans aide dans le dossier `~/workspace`. Vérifie ensuite que `git --version` fonctionne toujours.

Solution - A consulter après 20 min 
cd ~/workspace
pwd
git --version

Tu dois voir le chemin `~/workspace` puis une version Git. Si ce n'est pas le cas, c'est que tu n'es pas revenue dans le bon terminal ou le bon dossier.

Questions théoriques

Question 1 Pourquoi travaille-t-on dans le système de fichiers Linux de WSL plutôt que dans `C:\` ?

Réponse attendue

Parce que les outils Unix, Node, Java et les watchers de fichiers sont plus fiables et plus rapides dans le système de fichiers Linux natif de WSL que sur un montage Windows accessible via `/mnt/c`.

Question 2 Quelle est la différence entre “ouvrir Ubuntu” et “ouvrir PowerShell” dans cette phase ?

Réponse attendue

Ubuntu dans WSL exécute des commandes Unix dans un environnement Linux. PowerShell exécute des commandes Windows avec d'autres conventions de chemins, de shell et d'outils. Ici tout le guide suppose un environnement Unix.

Question 3 Pourquoi vérifie-t-on les outils avant de créer le repo ?

Réponse attendue

Parce qu'un setup explicite réduit les causes d'échec. Si une commande ne marche pas dès le départ, il vaut mieux le découvrir avant d'avoir déjà généré plusieurs projets et douter de la source du problème.

Definition of Done de l'étape

  • Tu sais ouvrir le bon terminal.
  • Le dossier `~/workspace` existe.
  • Tu sais vérifier où tu es avec pwd.
  • Tu as identifié si Git, Curl et Unzip sont installés.

Checklist manuelle

  • J'ai ouvert Ubuntu dans WSL.
  • Je travaille dans `~/workspace` et non dans `/mnt/c/...`.
  • `git --version` répond sans erreur.
  • `curl --version` répond sans erreur.
  • `unzip -v` répond sans erreur.
3

Créer le dossier du projet et initialiser Git

Objectif
Créer la racine du projet KataSensei et enlever toute ambiguïté sur l'existence du repo.

Pourquoi cette étape existe maintenant
Avant de générer un backend et un frontend, il faut savoir exactement où ils vont vivre et à partir de quel dossier les commandes doivent être lancées.

État de départ exact
Tu es dans `~/workspace`. Aucun dossier `katasensei` n'existe encore, ou s'il existe tu as décidé de repartir de zéro et tu l'as supprimé manuellement avant cette étape.

Micro-étapes

  1. Depuis le terminal Ubuntu ouvert dans `~/workspace`, crée le dossier racine du projet puis entre dedans : 
    mkdir katasensei
cd katasensei
    Résultat attendu : pwd doit retourner /home/ton-user/workspace/katasensei.
  2. Initialise Git dans ce dossier : 
    git init
    Résultat attendu : Git affiche un message indiquant qu'un dépôt vide a été créé.
  3. Force le nom de branche principale à `main` pour partir sur une base cohérente : 
    git branch -M main
    Vérification : 
    git branch --show-current
    Résultat attendu : main.
  4. Crée un `.gitignore` minimal à la racine du repo. Ouvre le dossier `~/workspace/katasensei` dans ton éditeur puis crée le fichier `.gitignore` avec ce contenu : 
    # Systèmes
.DS_Store
*.log

# Java
target
*.class
*.jar

# Node
node_modules
dist

# Environnement local
.env

# IDE
.idea
.vscode
  5. Crée aussi un fichier `README.md` minimal pour documenter le repo dès le départ : 
    # KataSensei

Monorepo initial du projet fil rouge.
  6. Vérifie l'état du repo : 
    git status --short
    Résultat attendu : tu dois voir `.gitignore` et `README.md` en fichiers non suivis.

Résultat attendu

Tu as maintenant un repo Git concret, avec une racine de projet explicite. Toutes les étapes suivantes partiront de `~/workspace/katasensei`.

Erreurs fréquentes

  • mkdir: cannot create directory 'katasensei': File exists : supprime le dossier `~/workspace/katasensei` créé par un essai précédent, puis recrée-le pour repartir sur une base propre.
  • git init exécuté dans le mauvais dossier : affiche `pwd`, place-toi dans `~/workspace/katasensei`, puis supprime le dossier `.git` créé au mauvais endroit avant de relancer `git init` au bon emplacement.
  • Le nom de branche n'est pas `main` : relance git branch -M main depuis la racine du repo.

Exercice

Ajoute dans `README.md` une phrase qui explique en une ligne le rôle de `katasensei-back` et `katasensei-front`, même si ces dossiers n'existent pas encore.

Solution - A consulter après 20 min 
# KataSensei

Monorepo initial du projet fil rouge.

- `katasensei-back` contiendra l'application Spring Boot.
- `katasensei-front` contiendra l'application Vue 3.

Questions théoriques

Question 1 Pourquoi initialise-t-on Git avant de générer les projets ?

Réponse attendue

Parce que cela fixe un périmètre clair dès le départ, permet de voir immédiatement les fichiers créés et évite de se demander plus tard ce qui appartient ou non au repo.

Question 2 Pourquoi avoir une branche principale `main` dès la phase 0 ?

Réponse attendue

Parce que la branche principale sert de référence commune pour la suite du workflow Git, les futures branches de travail et les futures automatisations.

Question 3 À quoi sert `.gitignore` à ce stade ?

Réponse attendue

Il évite de commiter des fichiers générés localement, lourds ou spécifiques à une machine, comme `node_modules`, `target` ou les fichiers d'IDE.

Definition of Done de l'étape

  • Le dossier `~/workspace/katasensei` existe.
  • Ce dossier est un repo Git.
  • La branche principale s'appelle `main`.
  • `.gitignore` et `README.md` existent.

Checklist manuelle

  • `pwd` affiche le dossier `~/workspace/katasensei`.
  • `git branch --show-current` retourne `main`.
  • `.gitignore` existe à la racine du repo.
  • `README.md` existe à la racine du repo.
Commit

C'est le bon moment pour commit : la racine du projet existe et le repo Git est initialisé proprement.

Fichiers à commit :

  • .gitignore
  • README.md

Ne commit pas encore les futurs dossiers katasensei-back et katasensei-front s'ils n'existent pas encore ou ne font pas partie de cette étape.

Message suggéré : chore(repo): initialisation du monorepo KataSensei

4

Installer Java 21 puis générer un backend Spring Boot minimal

Objectif
Obtenir un backend Java capable de démarrer localement dans le repo que tu viens de créer.

Pourquoi cette étape existe maintenant
La phase 0 doit prouver qu'un backend existe réellement. On installe donc Java et Maven avant de générer l'application Spring Boot.

État de départ exact
Tu es dans `~/workspace/katasensei`. Le repo existe, mais aucun dossier `katasensei-back` n'existe encore.

Micro-étapes

  1. Depuis le terminal Ubuntu ouvert à la racine du repo, installe SDKMAN : 
    curl -s "https://get.sdkman.io" | bash
source "$HOME/.sdkman/bin/sdkman-init.sh"
    Résultat attendu : la commande sdk version doit fonctionner.
  2. Installe Java 21 et Maven avec SDKMAN : 
    sdk install java 21.0.7-tem
sdk default java 21.0.7-tem
sdk install maven
    Vérification : 
    java -version
mvn -version
    Résultat attendu : Java 21 et Maven s'affichent sans erreur.
  3. Toujours depuis `~/workspace/katasensei`, télécharge le squelette Spring Boot directement dans `katasensei-back` : 
    curl -L "https://start.spring.io/starter.zip?type=maven-project&language=java&baseDir=katasensei-back&groupId=dev.katasensei&artifactId=katasensei-back&name=katasensei-back&description=Backend%20minimal%20de%20KataSensei&packageName=dev.katasensei.katasenseiback&packaging=jar&javaVersion=21&dependencies=web,actuator" -o katasensei-back.zip
unzip katasensei-back.zip
rm katasensei-back.zip
    Résultat attendu : le dossier `katasensei-back` existe à la racine du repo.
  4. Entre dans le dossier backend : 
    cd ~/workspace/katasensei/katasensei-back
  5. Lance l'application générée pour vérifier qu'elle démarre avant toute modification : 
    ./mvnw spring-boot:run
    Résultat attendu : tu dois voir dans les logs une ligne proche de Started KatasenseiBackApplication.
  6. Laisse ce terminal ouvert quelques secondes pour observer les logs, puis arrête l'application avec Ctrl + C.

Résultat attendu

Le backend Spring Boot existe, compile et démarre. Tu sais dans quel dossier il se trouve et avec quelle commande le lancer.

Erreurs fréquentes

  • sdk: command not found : le shell n'a pas été rechargé. Lance source "$HOME/.sdkman/bin/sdkman-init.sh".
  • java: command not found après installation : ferme puis rouvre le terminal Ubuntu, ou recharge le shell SDKMAN.
  • ./mvnw: Permission denied : exécute chmod +x mvnw puis relance la commande.
  • Le port 8080 est déjà utilisé : identifie le processus qui l'occupe ou arrête l'autre application avant de relancer le backend.

Exercice

Relance seule le backend une deuxième fois, puis arrête-le proprement avec Ctrl + C. L'objectif est d'ancrer la commande et le dossier à utiliser.

Solution - A consulter après 20 min 
cd ~/workspace/katasensei/katasensei-back
./mvnw spring-boot:run

Attends la ligne de démarrage, puis interrompt le process avec Ctrl + C.

Questions théoriques

Question 1 Pourquoi utilise-t-on SDKMAN pour Java et Maven ?

Réponse attendue

Parce que SDKMAN permet de gérer proprement plusieurs versions de Java et d'outils liés, sans dépendre du système global ni bricoler les chemins à la main.

Question 2 À quoi sert Spring Initializr ici ?

Réponse attendue

Spring Initializr génère un squelette de projet cohérent avec les bonnes dépendances, une structure standard et un point d'entrée prêt à lancer.

Question 3 Pourquoi vérifie-t-on d'abord que l'application générée démarre sans modification ?

Réponse attendue

Parce que cela permet d'isoler les problèmes. Si le projet généré ne démarre pas déjà, le souci vient du setup ou du scaffold, pas de ton code métier.

Definition of Done de l'étape

  • Java 21 est installé et visible avec java -version.
  • Maven est installé et visible avec mvn -version.
  • Le dossier `katasensei-back` existe.
  • Le backend démarre avec ./mvnw spring-boot:run.

Checklist manuelle

  • `java -version` répond avec Java 21.
  • `mvn -version` répond sans erreur.
  • Le dossier `katasensei-back` existe dans le repo.
  • J'ai vu le log de démarrage Spring Boot.
Commit

C'est le bon moment pour commit : le backend minimal existe et démarre sans modification métier supplémentaire.

Fichiers à commit :

  • katasensei-back/ dans l'état généré à cette étape

Ne commit pas encore les modifications de PingController.java ou de application.properties si elles appartiennent à l'étape suivante.

Message suggéré : chore(back): initialisation du backend Spring Boot minimal

5

Ajouter une preuve backend observable avec un health check et un endpoint simple

Objectif
Transformer le backend généré en un backend observable, avec une réponse HTTP testable depuis le terminal et le navigateur.

Pourquoi cette étape existe maintenant
Un projet qui “démarre” n'est pas encore une preuve suffisante. Il faut une sortie observable : un health check et une première route applicative.

État de départ exact
Le dossier `katasensei-back` existe et démarre. Tu es prête à modifier le code généré dans ton éditeur, tout en continuant à lancer les commandes depuis le terminal Ubuntu.

Micro-étapes

  1. Ouvre le dossier `~/workspace/katasensei` dans ton éditeur.
  2. Dans le backend, crée le fichier `src/main/java/dev/katasensei/katasenseiback/PingController.java` avec ce contenu : 
    package dev.katasensei.katasenseiback;

import java.util.Map;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class PingController {

    @GetMapping("/api/ping")
    public Map<String, String> ping() {
        return Map.of("message", "KataSensei back is alive");
    }
}
    Pourquoi ici : cette route sera la preuve la plus simple de communication front/back.
  3. Crée ensuite le fichier `src/main/java/dev/katasensei/katasenseiback/CorsConfig.java` pour centraliser CORS dès le début : 
    package dev.katasensei.katasenseiback;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class CorsConfig implements WebMvcConfigurer {

    private final String corsAllowedOrigin;

    public CorsConfig(@Value("${CORS_ALLOWED_ORIGIN}") String corsAllowedOrigin) {
        this.corsAllowedOrigin = corsAllowedOrigin;
    }

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins(corsAllowedOrigin)
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS");
    }
}
    Pourquoi ici : on évite d'éparpiller CORS sur chaque controller et on garde une seule origine autorisée, pilotée par l'environnement.
  4. Ouvre le fichier `src/main/resources/application.properties` et remplace son contenu par : 
    spring.application.name=katasensei-back
management.endpoints.web.exposure.include=health
    Résultat attendu : l'endpoint Actuator `health` devient exposé en HTTP.
  5. Dans un terminal Ubuntu placé dans `~/workspace/katasensei/katasensei-back`, définis d'abord l'origine frontend autorisée puis relance le backend : 
    export CORS_ALLOWED_ORIGIN=http://localhost:5173
./mvnw spring-boot:run
  6. Dans un second terminal Ubuntu, teste le health check : 
    curl http://localhost:8080/actuator/health
    Résultat attendu : une réponse JSON contenant un statut `UP`.
  7. Toujours dans le second terminal, teste l'endpoint métier minimal : 
    curl http://localhost:8080/api/ping
    Résultat attendu : 
    {"message":"KataSensei back is alive"}

Résultat attendu

Le backend ne se contente plus de démarrer. Il possède maintenant une route technique de santé et une route applicative minimale.

Erreurs fréquentes

  • 404 sur /actuator/health : vérifie que la propriété management.endpoints.web.exposure.include=health est bien présente.
  • 404 sur /api/ping : vérifie le package, le nom du fichier et les annotations @RestController et @GetMapping.
  • Port 8080 was already in use : un ancien backend tourne encore. Arrête-le avant de relancer.

Exercice

Modifie temporairement le message de l'endpoint `/api/ping` en `KataSensei backend ready`, relance le backend, puis vérifie la nouvelle réponse avec `curl`.

Solution - A consulter après 20 min

Dans `PingController.java`, change la valeur retournée :

return Map.of("message", "KataSensei backend ready");

Puis relance :

cd ~/workspace/katasensei/katasensei-back
./mvnw spring-boot:run

Ensuite :

curl http://localhost:8080/api/ping

Questions théoriques

Question 1 À quoi sert un health check dans un projet web ?

Réponse attendue

Il sert à vérifier rapidement si l'application est démarrée et capable de répondre, localement comme plus tard en déploiement ou en supervision.

Question 2 Pourquoi ajoute-t-on un endpoint `/api/ping` alors que le health check existe déjà ?

Réponse attendue

Parce que le health check est une preuve technique, alors que `/api/ping` est la première route applicative simple que le frontend pourra réellement appeler.

Question 3 Pourquoi configure-t-on déjà CORS globalement dès cette phase ?

Réponse attendue

Parce que le frontend Vite tournera sur un autre port que le backend. Sans configuration CORS minimale, le navigateur bloquera l'appel HTTP entre `localhost:5173` et `localhost:8080`. Le faire globalement évite de dupliquer `@CrossOrigin` dans chaque controller et prépare déjà le passage du local au déploiement.

  • `curl http://localhost:8080/actuator/health` répond avec `UP`.
  • `curl http://localhost:8080/api/ping` répond avec un JSON.
  • Je comprends pourquoi `CORS_ALLOWED_ORIGIN` pilote la configuration CORS globale du backend.

7

Faire communiquer le frontend et le backend

Objectif
Afficher dans le navigateur une donnée renvoyée par le backend pour prouver que le système complet fonctionne déjà.

Pourquoi cette étape existe maintenant
La phase 0 ne doit pas se terminer avec deux applications isolées. Il faut une preuve observable de communication front/back.

État de départ exact
Le backend répond sur `localhost:8080/api/ping`. Le frontend tourne sur `localhost:5173`. Les deux applications sont encore très simples.

Micro-étapes

  1. Laisse le backend démarré dans un premier terminal Ubuntu : 
    cd ~/workspace/katasensei/katasensei-back
./mvnw spring-boot:run
  2. Laisse le frontend démarré dans un second terminal Ubuntu : 
    cd ~/workspace/katasensei/katasensei-front
npm run dev
  3. Dans `katasensei-front/src/App.vue`, remplace le contenu par : 
    <script setup lang="ts">
import { onMounted, ref } from 'vue'

const backendMessage = ref('Chargement en cours...')
const errorMessage = ref('')

onMounted(async () => {
  try {
    const response = await fetch('http://localhost:8080/api/ping')

    if (!response.ok) {
      throw new Error('Le backend a répondu avec une erreur HTTP')
    }

    const data = (await response.json()) as { message: string }
    backendMessage.value = data.message
  } catch (error) {
    errorMessage.value =
      error instanceof Error ? error.message : 'Erreur inconnue pendant l'appel API'
  }
})
</script>

<template>
  <main>
    <h1>KataSensei démarre correctement</h1>
    <p>Frontend : visible sur le port 5173.</p>
    <p>Backend : attendu sur le port 8080.</p>
    <p>Message du backend : {{ backendMessage }}</p>
    <p v-if="errorMessage">Erreur observée : {{ errorMessage }}</p>
  </main>
</template>

<style scoped>
main {
  max-width: 720px;
  margin: 4rem auto;
  padding: 0 1rem;
  font-family: sans-serif;
}
</style>
  4. Retourne sur le navigateur à l'adresse du frontend et recharge la page.
  5. Résultat attendu : la ligne “Message du backend” doit afficher KataSensei back is alive ou la variante que tu as définie dans l'exercice précédent.
  6. Arrête ensuite l'un des deux serveurs pour observer un cas d'erreur. Par exemple, coupe le backend avec Ctrl + C puis recharge la page front. Résultat attendu : un message d'erreur apparaît.

Résultat attendu

Tu disposes maintenant de la première preuve complète du système : un front visible qui appelle un back réel et affiche sa réponse.

Erreurs fréquentes

  • Le navigateur affiche une erreur CORS : vérifie que `CorsConfig` existe bien et que le backend a été relancé avec CORS_ALLOWED_ORIGIN=http://localhost:5173.
  • Le message reste sur “Chargement en cours...” : vérifie que le backend tourne réellement sur le port 8080 et que l'URL appelée est correcte.
  • Le message d'erreur apparaît alors que le backend tourne : teste d'abord l'endpoint avec curl http://localhost:8080/api/ping pour savoir si le problème vient du backend ou du frontend.

Exercice

Change le texte du titre principal du frontend pour qu'il résume en une phrase ce que prouve cette phase 0.

Solution - A consulter après 20 min

Exemple de titre acceptable :

<h1>Le squelette fullstack KataSensei fonctionne en local</h1>

Le point important est que le titre parle d'une preuve réelle, pas d'une intention abstraite.

Réponse attendue

Réponse attendue

Installer flyctl dans WSL

  1. Dans un terminal Ubuntu, installe flyctl : 
    curl -L https://fly.io/install.sh | sh
    Résultat attendu : le script installe flyctl dans ~/.fly/bin/.
  2. Ajoute flyctl au PATH si ce n'est pas fait automatiquement. Ouvre ~/.zshrc et ajoute en fin de fichier : 
    export FLYCTL_INSTALL="$HOME/.fly"
export PATH="$FLYCTL_INSTALL/bin:$PATH"
    Puis recharge le shell : 
    source ~/.zshrc
  3. Vérifie l'installation : 
    fly version
    Résultat attendu : la version de flyctl s'affiche.
  4. Connecte-toi à ton compte Fly.io (crée-en un sur fly.io si besoin) : 
    fly auth login
    Résultat attendu : ton navigateur s'ouvre, tu te connectes, puis le terminal confirme l'authentification.

Préparer le déploiement du backend

  1. Crée le fichier katasensei-back/Dockerfile avec ce contenu : 
    FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY target/katasensei-back-0.0.1-SNAPSHOT.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
  2. Depuis le dossier backend, compile le jar avant de déployer : 
    cd ~/workspace/katasensei/katasensei-back
./mvnw package -DskipTests
    Résultat attendu : un fichier target/katasensei-back-0.0.1-SNAPSHOT.jar est créé.
  3. Initialise l'application Fly.io sans déployer immédiatement : 
    fly launch --name katasensei-back --no-deploy
    Résultat attendu : un fichier fly.toml est créé dans le dossier backend.
  4. Déclare ensuite l'origine frontend autorisée côté backend : 
    fly secrets set CORS_ALLOWED_ORIGIN=https://katasensei-front.fly.dev
    Résultat attendu : Fly.io enregistre le secret qui servira à la configuration CORS globale du backend.
  5. Déploie le backend : 
    fly deploy
    Résultat attendu : Fly.io construit l'image Docker, la déploie et affiche l'URL publique de l'application.
  6. Vérifie le health check en production : 
    curl https://katasensei-back.fly.dev/actuator/health
    Résultat attendu : réponse JSON avec un statut UP.

Préparer le déploiement du frontend

  1. Crée le fichier katasensei-front/nginx.conf pour servir les fichiers statiques Vite : 
    server {
  listen 8080;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }
}
  2. Crée le fichier katasensei-front/Dockerfile : 
    FROM node:22-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 8080
  3. Dans katasensei-front/src/App.vue, remplace l'URL codée en dur par une variable d'environnement Vite. Dans le <script setup>, change : 
    const response = await fetch('http://localhost:8080/api/ping')
    par : 
    const response = await fetch(`${import.meta.env.VITE_API_URL}/api/ping`)
    Puis crée le fichier katasensei-front/.env pour le développement local : 
    VITE_API_URL=http://localhost:8080
    Ce fichier est dans .gitignore. Crée aussi katasensei-front/.env.example avec une valeur fictive à committer : 
    VITE_API_URL=https://ton-backend.fly.dev
  4. Depuis le dossier frontend, initialise l'application Fly.io sans déployer : 
    cd ~/workspace/katasensei/katasensei-front
fly launch --name katasensei-front --no-deploy
  5. Injecte la variable d'environnement pointant vers le backend déployé : 
    fly secrets set VITE_API_URL=https://katasensei-back.fly.dev
  6. Déploie le frontend : 
    fly deploy
    Résultat attendu : Fly.io construit l'image, déploie et affiche l'URL publique du frontend.

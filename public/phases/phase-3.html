Phase 3

PrimeVue, composants métier du catalogue, LogService et Stylelint

Durée estimée : ~10h à ~12h réparties sur 2 semaines

La phase 2 a sécurisé le flux de données du catalogue. La phase 3 transforme ce socle en une vraie interface produit : une vue lisible, des états d'écran clairs, des composants métier front, une journalisation centralisée et un CSS enfin contrôlé par un linter.

🟠UI métier 🟢Composants 🟢Architecture front 🟣Qualité

Objectif de phase

Refondre l'affichage du catalogue de katas avec `PrimeVue`, introduire des composants métier front, centraliser les logs via `LogService`, puis encadrer le CSS avec `Stylelint`.

État de départ exact

  • Tu travailles sur Windows 11, mais toutes les commandes du projet se font dans Ubuntu via WSL2.
  • La phase 2 est terminée et le frontend possède déjà `KataRepository`, `GetKatasUseCase`, `useKataStore` et `useKataList`.
  • Le catalogue s'affiche déjà, mais il reste textuel et peu rassurant visuellement.
  • Les états `loading`, `error`, `success` et `idle` existent déjà dans le flux front.
  • Le frontend n'utilise pas encore `PrimeVue`.
  • Le frontend n'a pas encore de `LogService` ni de `Stylelint`.

Résultat attendu en fin de phase

  • `PrimeVue` est installé et configuré dans le frontend.
  • Le catalogue affiche un vrai titre, des cartes de katas lisibles et des badges de niveau.
  • Le catalogue possède un composant `KataListState` pour `loading`, `empty` et `error`.
  • `GetKatasUseCase` journalise via `LogService` au lieu de dépendre de `console` partout.
  • `Stylelint` contrôle les styles `.vue`, `.css` et `.scss` du frontend.
  • Le frontend est validé localement puis redéployé sur son environnement public.

Valeur livrée à la personne qui utilise KataSensei

Le catalogue devient plus lisible, plus rassurant et plus cohérent. Il reste utile même pendant le chargement ou en cas d'erreur, au lieu de retomber sur un rendu brut de développement.

Prérequis

  • La phase 2 est terminée et le catalogue se charge encore localement.
  • Tu sais relancer le backend et le frontend depuis Ubuntu dans WSL.
  • Tu sais déjà lancer `npm run type-check`, `npm run test` et `npm run build` dans le frontend.

Outils mobilisés

  • Terminal Ubuntu dans WSL
  • VS Code
  • Vue 3
  • TypeScript strict
  • Pinia
  • Vitest et Vue Test Utils
  • PrimeVue
  • Stylelint
  • Fly.io pour le redéploiement frontend
Info parcours
Cette phase reste centrée sur le catalogue déjà construit. On n'ajoute pas un nouveau flux métier. On améliore l'interface, les composants front, la stratégie de logs et la qualité CSS sur une fonctionnalité déjà vivante.
1

Repartir du catalogue actuel

Objectif
Relancer le catalogue hérité de la phase 2, vérifier ses états exacts et poser une première structure d'écran visible.

Pourquoi cette étape existe maintenant
La phase 3 doit améliorer un écran existant, pas partir d'une page imaginaire. Il faut donc repartir du catalogue réel, puis lui donner immédiatement une structure de page plus claire.

État de départ exact
Le flux du catalogue fonctionne déjà, mais la vue reste brute et la hiérarchie visuelle de l'écran est encore faible.

Avant de commencer : recherche de logs

Dans VS Code, la recherche globale Ctrl+Shift+F avec console. suffit pour repérer les logs directs.

En terminal, la commande ci-dessous fait la même chose. `rg` signifie "ripgrep" et -n affiche les numéros de ligne :

rg -n "console\\.(log|info|warn|error|debug)" src tests

Étapes

  3. Ouvre un second terminal Ubuntu et démarre le backend :

Erreurs fréquentes

  • Ne commence pas par installer `PrimeVue` si le catalogue ne s'affiche plus. Corrige d'abord le point de départ.
  • Ne change pas le nom des statuts hérités de la phase 2. Cette phase améliore l'UI, elle ne redéfinit pas le contrat de `AsyncState`.
  • Ne supprime pas le contenu existant de la vue tant que le frontend n'affiche pas encore ton nouvel en-tête correctement.

Exercice

Dans `KataListView.vue`, traite `idle` comme le premier chargement utile de l'écran. Remplace le message hérité de phase 2 par le même rendu que `loading` :

<p v-if="state.status === 'loading' || state.status === 'idle'" role="status">
  Chargement du catalogue...
</p>
Solution - A consulter après 20 min

Le bon comportement en phase 3 est de traiter `idle` comme une attente de premier chargement, pas comme un cinquième état d'écran à part.

Le remplacement utile est donc bien :

<p v-if="state.status === 'loading' || state.status === 'idle'" role="status">
  Chargement du catalogue...
</p>

Questions théoriques

1. Pourquoi repartir du catalogue actuel au lieu de créer directement des composants UI nouveaux ?

Réponse attendue

Parce que la phase 3 améliore une fonctionnalité déjà existante. Il faut d'abord relire ce qui est en place pour savoir précisément ce que l'on corrige.

2. Pourquoi garder exactement les statuts `idle`, `loading`, `error` et `success` ?

Réponse attendue

Parce que ce contrat existe déjà dans la phase 2. La phase 3 doit construire au-dessus de ce socle, pas réinventer son modèle d'état.

3. Pourquoi ajouter un vrai en-tête avant d'installer une bibliothèque UI ?

Réponse attendue

Parce qu'une meilleure interface commence par une meilleure structure d'écran. La hiérarchie visuelle ne dépend pas d'abord d'une bibliothèque, mais d'un rendu plus clair.

Installer PrimeVue et créer KataCard

Objectif
Installer `PrimeVue`, brancher le thème global, puis remplacer la liste brute du cas nominal par un vrai composant `KataCard`.

Pourquoi cette étape existe maintenant
Le problème concret est maintenant visible : la vue a une structure, mais le cas nominal reste trop brut. C'est le bon moment pour installer `PrimeVue` et améliorer d'abord ce rendu-là.

État de départ exact
Le catalogue possède maintenant une structure de page plus claire, mais il affiche encore sa liste de façon textuelle.

Étapes

  1. Dans un terminal Ubuntu placé dans `~/workspace/katasensei/katasensei-front`, installe `PrimeVue` et son preset de thème : 
    npm install primevue @primeuix/themes
    Résultat attendu : `primevue` et `@primeuix/themes` apparaissent dans `package.json`.
  2. Ouvre `src/main.ts` et ajoute exactement ces deux imports : 
    import PrimeVue from 'primevue/config'
import Aura from '@primeuix/themes/aura'
  3. Dans ce même fichier, juste avant `app.mount('#app')`, ajoute : 
    app.use(PrimeVue, {
  theme: {
    preset: Aura
  }
})
    Résultat attendu : `PrimeVue` est branché avec le thème `Aura`.
  4. Crée `src/ui/components/KataCard.vue` avec ce contenu complet : 
    <script setup lang="ts">
import Card from 'primevue/card'
import Tag from 'primevue/tag'
import type { Kata } from '@/domain/models/Kata'

defineProps<{
  kata: Kata
}>()

const severityByLevel = {
  debutant: 'success',
  intermediaire: 'warn',
  avance: 'danger'
} as const

const labelByLevel = {
  debutant: 'Débutant',
  intermediaire: 'Intermédiaire',
  avance: 'Avancé'
} as const
</script>

<template>
  <Card class="kata-card">
    <template #title>
      {{ kata.titre }}
    </template>

    <template #subtitle>
      <Tag
        :value="labelByLevel[kata.niveau]"
        :severity="severityByLevel[kata.niveau]"
        rounded
      />
    </template>

    <template #content>
      <p class="kata-card__description">
        {{ kata.description }}
      </p>
    </template>
  </Card>
</template>

<style scoped>
.kata-card {
  height: 100%;
}

.kata-card__description {
  margin: 0;
  line-height: 1.5;
}
</style>
  5. Ouvre ensuite `src/ui/views/KataListView.vue` et remplace la branche `success` textuelle par celle-ci :

Questions théoriques

1. Pourquoi installer `PrimeVue` maintenant et pas en phase 0 ?

Réponse attendue

Parce qu'une bibliothèque UI n'a de valeur que lorsqu'un écran concret existe déjà et a besoin d'un meilleur rendu. Avant cela, elle ajouterait surtout de la complexité prématurée.

2. Pourquoi commencer par améliorer le cas nominal avec `KataCard` ?

Réponse attendue

Parce que le cas nominal est le chemin principal de lecture du catalogue. L'améliorer d'abord rend immédiatement visible la valeur de la phase.

Réponse attendue

Parce qu'aucune duplication de configuration n'est encore prouvée. Il est plus simple et plus pédagogique de partir d'un composant métier clair que d'anticiper des wrappers locaux trop tôt.

3

Construire KataListState

Objectif
Remplacer les messages bruts du `loading`, du `empty` et du `error` par un vrai composant d'état d'écran.

Pourquoi cette étape existe maintenant
Le cas nominal est maintenant crédible. Il faut maintenant traiter sérieusement les cas dégradés, qui faisaient encore très "développement" dans la vue de phase 2.

État de départ exact
Le catalogue affiche déjà des cartes en cas de succès, mais les autres états sont encore rendus directement dans `KataListView.vue`.

Étapes

  1. Crée `src/ui/components/KataListState.vue` avec ce contenu complet : 
    <script setup lang="ts">
import Button from 'primevue/button'
import Message from 'primevue/message'
import Skeleton from 'primevue/skeleton'

withDefaults(
  defineProps<{
    status: 'loading' | 'empty' | 'error'
    message?: string
  }>(),
  {
    message: 'Le catalogue n’a pas pu être chargé.'
  }
)

const emit = defineEmits<{
  retry: []
}>()
</script>

<template>
  <div
    v-if="status === 'loading'"
    class="kata-list-state kata-list-state--loading"
    aria-live="polite"
  >
    <div
      v-for="index in 3"
      :key="index"
      class="kata-list-state__skeleton-card"
    >
      <Skeleton width="12rem" height="1.5rem" />
      <Skeleton width="7rem" height="1rem" />
      <Skeleton width="100%" height="4rem" />
    </div>
  </div>

  <Message
    v-else-if="status === 'empty'"
    severity="info"
    class="kata-list-state"
  >
    Aucun kata disponible pour le moment.
  </Message>

  <div v-else class="kata-list-state kata-list-state--error">
    <Message severity="error">
      {{ message }}
    </Message>

    <Button
      label="Réessayer"
      severity="secondary"
      @click="emit('retry')"
    />
  </div>
</template>

<style scoped>
.kata-list-state {
  display: grid;
  gap: 1rem;
}

.kata-list-state--loading {
  grid-template-columns: repeat(auto-fit, minmax(16rem, 1fr));
}

.kata-list-state__skeleton-card {
  display: grid;
  gap: 0.75rem;
  padding: 1rem;
  border: 1px solid var(--border-color, #dcdcdc);
  border-radius: 0.75rem;
}

.kata-list-state--error {
  justify-items: start;
}
</style>
  2. Ouvre `src/ui/views/KataListView.vue` et remplace les branches de rendu d'état par celles-ci : 
    <script setup lang="ts">
import { computed } from 'vue'
import KataCard from '@/ui/components/KataCard.vue'
import KataListState from '@/ui/components/KataListState.vue'
import { useKataList } from '@/ui/composables/useKataList'
import { useKataStore } from '@/stores/useKataStore'

const { state } = useKataList()
const kataStore = useKataStore()

const katas = computed(() => {
  return state.value.status === 'success' ? state.value.data : []
})

const errorMessage = computed(() => {
  return state.value.status === 'error'
    ? state.value.message
    : 'Le catalogue n’a pas pu être chargé.'
})

function rechargerCatalogue() {
  void kataStore.chargerKatas()
}
</script>

<template>
  <section class="kata-list-view">
    <header class="kata-list-view__header">
      <h1 class="kata-list-view__title">Catalogue de katas</h1>
      <p class="kata-list-view__subtitle">
        Choisis un kata et entraîne-toi sur une difficulté adaptée à ton niveau actuel.
      </p>
    </header>

    <KataListState
      v-if="state.status === 'loading' || state.status === 'idle'"
      status="loading"
    />

    <KataListState
      v-else-if="state.status === 'error'"
      status="error"
      :message="errorMessage"
      @retry="rechargerCatalogue"
    />

    <KataListState
      v-else-if="state.status === 'success' && katas.length === 0"
      status="empty"
    />

    <ul v-else class="kata-list-view__grid">
      <li
        v-for="kata in katas"
        :key="kata.id"
        class="kata-list-view__item"
      >
        <KataCard :kata="kata" />
      </li>
    </ul>
  </section>
</template>
    Résultat attendu : la vue n'affiche plus directement les messages d'état. Elle délègue maintenant tout cela à `KataListState`.
  3. Crée `tests/ui/components/KataListState.test.ts` : 
    import { mount } from '@vue/test-utils'
import { describe, expect, it } from 'vitest'
import KataListState from '@/ui/components/KataListState.vue'

describe('KataListState', () => {
  it('affiche un message vide explicite', () => {
    const wrapper = mount(KataListState, {
      props: { status: 'empty' }
    })

    expect(wrapper.text()).toContain('Aucun kata disponible')
  })

  it('affiche le message d erreur et le bouton Reessayer', () => {
    const wrapper = mount(KataListState, {
      props: {
        status: 'error',
        message: 'API hors ligne'
      }
    })

    expect(wrapper.text()).toContain('API hors ligne')
    expect(wrapper.text()).toContain('Réessayer')
  })

  it('affiche trois cartes squelette pendant le chargement', () => {
    const wrapper = mount(KataListState, {
      props: { status: 'loading' }
    })

    expect(wrapper.findAll('.kata-list-state__skeleton-card')).toHaveLength(3)
  })
})
  4. Lance les tests ciblés : 
    npm run test -- tests/ui/components/KataCard.test.ts tests/ui/components/KataListState.test.ts
    Résultat attendu : les deux fichiers passent.
  5. Recharge le navigateur. Résultat attendu : le cas nominal affiche les cartes, et les cas `loading`, `empty` et `error` passent tous par un composant d'état dédié.

Erreurs fréquentes

  • Ne remets pas de messages d'état directement dans `KataListView.vue` une fois `KataListState` créé.
  • Ne fais pas de `KataListState` un composant qui appelle l'API. Son rôle est uniquement d'afficher l'état d'écran et d'émettre `retry`.
  • Ne laisse pas le cas `success` avec liste vide tomber dans la même branche que la grille de cartes.

Exercice

Ajoute à `tests/ui/components/KataListState.test.ts` ce test qui prouve l'émission de `retry` :

it('emet retry quand on clique sur Reessayer', async () => {
  const wrapper = mount(KataListState, {
    props: {
      status: 'error',
      message: 'API hors ligne'
    }
  })

  await wrapper.get('button').trigger('click')

  expect(wrapper.emitted('retry')).toHaveLength(1)
})
Solution - A consulter après 20 min

Le test passe avec ce bouton :

<Button
  label="Réessayer"
  severity="secondary"
  @click="emit('retry')"
/>

Il prouve qu'un état d'erreur utile ne se contente pas d'afficher un problème : il offre aussi une action de reprise.

Questions théoriques

  feature: string
  action: string

}

export interface LogService {
  debug(message: string, context: LogContext): void

  2. Crée l'implémentation `src/infrastructure/logging/ConsoleLogService.ts` : 
    import type { LogContext, LogLevel, LogService } from '@/domain/ports/LogService'

const SENSITIVE_KEYS = ['token', 'authorization', 'password']

function sanitizeContext(context: LogContext) {
  return Object.fromEntries(
    Object.entries(context).map(([key, value]) => {
      if (SENSITIVE_KEYS.includes(key.toLowerCase())) {
        return [key, '[redacted]']
      }

      return [key, value]
    })
  )
}

export class ConsoleLogService implements LogService {
  constructor(private readonly environment: string = import.meta.env.MODE) {}

  debug(message: string, context: LogContext) {
    if (this.environment === 'production') {
      return
    }

    this.write('debug', message, context)
  }

  info(message: string, context: LogContext) {
    this.write('info', message, context)
  }

  warn(message: string, context: LogContext) {
    this.write('warn', message, context)
  }

  error(message: string, context: LogContext) {
    this.write('error', message, context)
  }

  private write(level: LogLevel, message: string, context: LogContext) {
    const payload = {
      level,
      message,
      timestamp: new Date().toISOString(),
      ...sanitizeContext(context)
    }

    const target =
      level === 'error'
        ? console.error
        : level === 'warn'
          ? console.warn
          : level === 'info'
            ? console.info
            : console.debug

    target(payload)
  }
}

  return new ConsoleLogService(import.meta.env.MODE)
}

# PHASE 1

Ce fichier décrit le cadrage spécifique de la phase 1.

## Rôle de la phase

La phase 1 doit transformer le squelette fullstack minimal de la phase 0 en premier incrément métier structuré, avec les premiers choix de qualité et de persistance réellement justifiés par le besoin.

## Point de départ supposé

- la phase 0 est validée
- WSL2, Ubuntu, `zsh` et Oh My Zsh sont installés
- le repo Git existe
- le backend Spring Boot démarre localement
- le frontend Vue démarre localement
- le frontend affiche déjà une réponse du backend

## Reports hérités obligatoires à relire

Relire `docs/NEXT_PHASE.md` avant toute proposition de structure cible.

Les reports déjà identifiés depuis la phase 0 sont notamment :

- outillage qualité minimal front
- persistance et variables d'environnement
- conventions de style et de tests
- CI/CD ou préparation au déploiement
- ADR quand une vraie décision structurante apparaît

## Travail attendu avant réécriture

- produire ou mettre à jour `docs/audits/PHASE_1_AUDIT.md` avant toute autre proposition
- faire de cet audit la première réponse de travail à l'utilisateur
- attendre validation de cet audit avant toute édition
- auditer l'existant de `site/phases/phase-1.html`
- vérifier ce qui existe déjà dans les contenus source utiles au scope de la phase 1
- reprendre les reports pertinents hérités de la phase 0
- proposer une structure cible complète avant toute édition

## À compléter avant ou pendant la phase 1

- préciser le livrable exact de fin de phase 1
- préciser le premier incrément métier réellement visé
- préciser les fichiers source internes à relire si nécessaire

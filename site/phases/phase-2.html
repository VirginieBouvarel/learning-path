Comprendre la structure cible avant de coder

Avant d'écrire une ligne, relis le code naïf de la phase 1 et réponds à ces questions :

  1. Combien d'endroits font un fetch directement dans un composant ou un composable ?
  2. Si l'API change le nom du champ titre en name, combien de fichiers dois-tu modifier ?
  3. Si tu veux tester useKataList sans navigateur, est-ce possible ? Pourquoi ?
  4. Si loading, error, data sont trois ref indépendants, que se passe-t-il si on oublie de mettre loading = false dans le catch ?

Ces quatre douleurs ont chacune une solution dans cette phase. Garde tes réponses — elles te serviront de motivation.

Ce qu'on va faire maintenant : on va découper le front en quatre zones claires. Le domain/ contiendra les types et la logique pure, infrastructure/ parlera à l'extérieur, stores/ gardera l'état partagé, et ui/ pilotera Vue. Le but n'est pas de "faire de la belle archi", mais de rendre le code testable, modifiable et lisible sans effet domino.

La structure cible

src/
├── domain/              ← Zéro import Vue, zéro Pinia — Vitest pur
│   ├── types/             (branded types, result, unions, guards)
│   ├── ports/             (interfaces — KataRepository, LogPort)
│   └── usecases/          (logique pure — GetKatasUseCase)
│
├── infrastructure/      ← Adapters + composables headless
│   ├── api/
│   │   ├── ApiKataRepository.ts
│   │   └── mappers/           (ApiKataDto → Kata — ACL)
│   ├── logging/
│   └── composables/       (useAsync — ref/watch ok, pas de onMounted)
│
├── stores/              ← Pinia — état seul, appelle les use cases
│
└── ui/
    ├── composables/       ← onMounted, useRouter, stores (cycle de vie Vue)
    ├── components/
    └── views/

Questions théoriques

Question 0.1 : pourquoi séparer domain et infrastructure au lieu de tout mettre dans src/ ?

Réponse attendue

Parce que le domaine doit survivre à un changement technique. Si demain l'API, Pinia ou Vue changent, la logique métier doit rester stable. La séparation rend les dépendances visibles et limite les régressions.

Question 0.2 : pourquoi les hooks Vue comme onMounted n'ont-ils rien à faire dans domain/ ?

Réponse attendue

Parce qu'ils dépendent du runtime Vue et du cycle de vie d'un composant. Le domaine doit rester exécutable et testable sans navigateur, sans composant monté et sans framework UI.

Question 0.3 : quel symptôme indique que l'API "contamine" trop le front ?

Réponse attendue

Quand un simple changement de payload backend oblige à modifier plusieurs composants, stores et tests. Cela signifie qu'il manque une couche de traduction centrale, donc un mapper ou un port bien placé.

commit suggestion → chore(front): mise en place structure domain/infra/stores/ui

Branded types

Exercice 1.1 — Crée src/domain/types/branded.ts

Objectif : empêcher TypeScript de confondre des identifiants qui ont la même forme (string) mais pas le même sens métier.

Procède dans cet ordre :

  1. Crée le fichier src/domain/types/branded.ts.
  2. Déclare d'abord les 3 types sans helper.
  3. Ajoute ensuite un helper de conversion par type.
  4. N'utilise as nulle part ailleurs dans le projet.
  5. Lance npm run type-check pour vérifier que rien n'est cassé.
  • KataId, UserId, TentativeId en branded types
  • Fonctions helpers toKataId(s: string): KataId — seuls endroits où as est autorisé
  • Ces helpers sont les uniques portes d'entrée pour créer des branded types

Question : pourquoi centraliser les as dans des helpers plutôt que de les écrire partout ?

Solution — ouvre seulement si bloquée après 20 min 
export type KataId    = string & { readonly _brand: 'KataId' }
export type UserId    = string & { readonly _brand: 'UserId' }
export type TentativeId = string & { readonly _brand: 'TentativeId' }

export const toKataId    = (s: string): KataId    => s as KataId
export const toUserId    = (s: string): UserId    => s as UserId
export const toTentativeId = (s: string): TentativeId => s as TentativeId

Réponse : un seul endroit à auditer si on veut valider les IDs. Si demain on ajoute une validation UUID, on la met dans toKataId et tout le projet en bénéficie.

Questions théoriques

Question 1.2 : pourquoi un branded type n'existe-t-il qu'à la compilation et pas à l'exécution ?

Réponse attendue

Parce que TypeScript efface ses types au runtime. Le branded type protège le code pendant l'écriture et la compilation, mais en JavaScript final cela reste une string classique.

Question 1.3 : à quel moment faut-il convertir une string brute en KataId ?

Réponse attendue

Au point d'entrée du système concerné : juste après la lecture d'une réponse API, d'une route ou d'un formulaire. On normalise tôt pour travailler ensuite avec des types métier sûrs.

commit suggestion → feat(typage): ajout branded types KataId UserId TentativeId

Discriminated unions

Exercice 2.1 — AsyncState<T> dans src/domain/types/async.ts

4 états : idle, loading, success<T>, error. Fonctions constructeurs pour chaque état.

Ce que tu dois faire concrètement :

  1. Crée le type union avec un discriminant commun nommé status.
  2. Vérifie que seul l'état success porte data.
  3. Vérifie que seul l'état error porte message.
  4. Ajoute ensuite les helpers idle(), loading(), success(data) et error(message).
Solution — ouvre seulement si bloquée après 20 min 
export type AsyncState<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; message: string }

export const idle = <T>(): AsyncState<T> => ({ status: 'idle' })
export const loading = <T>(): AsyncState<T> => ({ status: 'loading' })
export const success = <T>(data: T): AsyncState<T> => ({ status: 'success', data })
export const error = <T>(message: string): AsyncState<T> => ({ status: 'error', message })

Exercice 2.2 — TentativeState dans src/domain/types/tentative.ts

idle | running (avec startedAt, indicesUtilises) | completed (avec score, completedAt). Écris canRequestIndice(tentative: TentativeState): boolean avec le pattern never pour l'exhaustivité.

Méthode conseillée :

  1. Écris d'abord le type TentativeState.
  2. Ajoute ensuite la fonction canRequestIndice avec un switch.
  3. Ne mets le cas default qu'à la fin, avec la variable never.
  4. Teste mentalement le cas paused pour comprendre ce que protège le compilateur.

Question : que se passe-t-il si tu ajoutes un état paused sans mettre à jour canRequestIndice ?

Solution — ouvre seulement si bloquée après 20 min 
export function canRequestIndice(tentative: TentativeState): boolean {
  switch (tentative.status) {
    case 'idle':      return false
    case 'running':   return tentative.indicesUtilises < 3
    case 'completed': return false
    default: {
      const _exhaustive: never = tentative
      return _exhaustive
    }
  }
}

Réponse : TypeScript émet une erreur sur const _exhaustive: never = tentative — paused n'est pas assignable à never. Le compilateur te force à gérer chaque cas.

Questions théoriques

Question 2.2 : pourquoi préférer une union discriminée à trois ref séparées comme loading, error et data ?

Réponse attendue

Parce qu'une union modélise explicitement les états autorisés et interdit les combinaisons absurdes, par exemple "loading=true" et "data déjà présente" si ce n'est pas voulu. On remplace une convention fragile par un contrat de type.

Question 2.3 : à quoi sert exactement le discriminant status ?

Réponse attendue

Il permet à TypeScript de raffiner automatiquement le type dans chaque branche du switch. Grâce à lui, TS sait qu'en cas de success, data existe, et qu'en cas de error, c'est message qui existe.

Question 2.4 : pourquoi le pattern never est-il précieux dans une base TypeScript stricte ?

Réponse attendue

Parce qu'il transforme l'oubli d'un cas en erreur de compilation immédiate. On ne dépend plus d'un test manuel tardif ou d'un bug en production pour découvrir qu'un nouvel état n'est pas géré.

commit suggestion → feat(typage): ajout discriminated unions AsyncState et TentativeState

Result pattern + AppError

Exercice 3.1 — Result<T,E> dans src/domain/types/result.ts

Union ok: true / value: T ou ok: false / error: E. Fonctions ok(), err(), map(), mapError().

Pas à pas : écris d'abord le type, puis les helpers ok/err, puis seulement les fonctions de transformation map et mapError. Vérifie après chaque ajout que l'inférence de type reste correcte.

Solution — ouvre seulement si bloquée après 20 min 
export type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E }

export const ok = <T>(value: T): Result<T, never> => ({ ok: true, value })
export const err = <E>(error: E): Result<never, E> => ({ ok: false, error })

export function map<T, U, E>(result: Result<T, E>, fn: (value: T) => U): Result<U, E> {
  return result.ok ? ok(fn(result.value)) : result
}

Exercice 3.2 — AppError dans src/domain/types/errors.ts

NetworkError | NotFoundError | ValidationError | UnauthorizedError | UnexpectedError. Fonction toUserMessage(error: AppError): string exhaustive avec never.

Solution — ouvre seulement si bloquée après 20 min 
export type Result<T, E> =
  | { ok: true;  value: T }
  | { ok: false; error: E }

export const ok  = <T>(value: T): Result<T, never>  => ({ ok: true, value })
export const err = <E>(error: E): Result<never, E>  => ({ ok: false, error })

export function map<T, U, E>(result: Result<T, E>, fn: (v: T) => U): Result<U, E> {
  return result.ok ? ok(fn(result.value)) : result
}

Questions théoriques

Question 3.1 : pourquoi préférer un Result à des throw dans le front ?

Réponse attendue

Parce que le type signale explicitement qu'une opération peut échouer. L'appelant est forcé de traiter les deux branches et la circulation de l'erreur devient lisible au lieu de dépendre d'une exception implicite.

Question 3.2 : quelle différence entre une erreur métier typée et un simple message texte ?

Réponse attendue

Une erreur typée peut être branchée, testée et convertie de façon fiable. Un simple texte n'est ni stable ni robuste pour piloter du comportement applicatif.

Question 3.3 : pourquoi garder toUserMessage exhaustive ?

Réponse attendue

Pour qu'un nouveau type d'erreur oblige le développeur à décider immédiatement quel message montrer. Sans ça, certaines erreurs deviennent muettes ou retombent dans un message générique trompeur.

commit suggestion → feat(typage): ajout Result pattern et AppError union
4

Type guards + template literal types

Exercice 4.1 — Type guards dans src/domain/types/guards.ts

isNetworkError, isValidationError, isAppError — chacun vérifie la forme à runtime. Écris des tests Vitest pour chacun.

Question : différence entre as NetworkError et un type guard ?

Solution — ouvre seulement si bloquée après 20 min 
export function isNetworkError(error: unknown): error is NetworkError {
  return typeof error === 'object'
    && error !== null
    && 'kind' in error
    && (error as { kind?: unknown }).kind === 'network'
}

export function isAppError(error: unknown): error is AppError {
  return typeof error === 'object'
    && error !== null
    && 'kind' in error
}

Réponse : as force le compilateur à te croire sans aucune vérification. Un type guard, lui, vérifie réellement la forme à l'exécution et informe ensuite TypeScript que le raffinement est légitime.

Exercice 4.2 — Routes typées dans src/infrastructure/api/routes.ts

Type KataRoute = `/katas/${KataId}`. Fonction buildKataRoute(id: KataId): KataRoute. Il doit être impossible d'appeler la fonction avec un string brut.

Solution — ouvre seulement si bloquée après 20 min 
export type KataRoute = `/katas/${KataId}`

export function buildKataRoute(id: KataId): KataRoute {
  return `/katas/${id}` as KataRoute
}

Questions théoriques

Question 4.2 : pourquoi un type guard est-il surtout utile sur les données venant de l'extérieur ?

Réponse attendue

Parce que ces données ne sont pas garanties par le compilateur : réseau, localStorage, params d'URL, JSON arbitraire. C'est précisément là qu'il faut réintroduire des vérifications runtime.

Question 4.3 : qu'apporte un template literal type sur une route ?

Réponse attendue

Il encode la forme attendue directement dans le type. Cela réduit les concaténations hasardeuses et documente immédiatement le contrat de l'URL.

commit suggestion → feat(typage): ajout type guards et routes typées

LogService

Exercice 5.1 — Port dans src/domain/ports/LogPort.ts

Interface avec debug, info, warn, error — chacun prend context: string et data?: Record<string, unknown>. C'est une interface dans domain/ — elle ne sait pas si les logs vont dans la console ou dans Sentry.

Solution — ouvre seulement si bloquée après 20 min 
export interface LogPort {
  debug(context: string, data?: Record<string, unknown>): void
  info(context: string, data?: Record<string, unknown>): void
  warn(context: string, data?: Record<string, unknown>): void
  error(context: string, data?: Record<string, unknown>): void
}

Exercice 5.2 — Adapters dans src/infrastructure/logging/

ConsoleLogger (logs colorés en dev) et SilentLogger (ne fait rien, pour les tests). Export selon l'environnement dans index.ts.

Question : pourquoi SilentLogger plutôt que de mocker console dans les tests ?

Solution — ouvre seulement si bloquée après 20 min 
export class SilentLogger implements LogPort {
  debug(): void {}
  info(): void {}
  warn(): void {}
  error(): void {}
}

Réponse : parce qu'un faux adapter stable est plus simple, plus lisible et plus proche de l'architecture réelle qu'un mock global de console. On teste l'injection d'une dépendance, pas le runtime du navigateur.

Questions théoriques

Question 5.2 : pourquoi le logger est-il un port et pas un utilitaire global ?

Réponse attendue

Parce que le domaine ne doit pas être couplé à un mécanisme technique précis. Un port permet de remplacer la console par Sentry, Datadog ou rien du tout sans modifier la logique appelante.

Question 5.3 : pourquoi éviter d'écrire directement console.log dans les use cases ?

Réponse attendue

Parce que cela introduit un effet de bord technique dans une couche qui doit rester pure et testable. Le code métier devient alors plus difficile à isoler et à vérifier.

commit suggestion → feat(infra): ajout LogPort ConsoleLogger SilentLogger

Mappers (Anti-Corruption Layer)

C'est l'étape la plus importante pour l'archi hexa front. Les mappers traduisent entre le langage de l'API et le langage du domaine. Sans eux, les types de l'API contaminent le domaine.

Exercice 6.1 — Définis le type de réponse API

Crée src/infrastructure/api/dto/ApiKataDto.ts qui représente exactement ce que l'API renvoie.

  • id: string (pas encore KataId — c'est le rôle du mapper de convertir)
  • titre: string
  • consigne: string
  • niveau: string (pas encore l'enum)
Ce type représente la réalité de l'API. Si l'API change, seul ce fichier change.
Solution — ouvre seulement si bloquée après 20 min 
export interface ApiKataDto {
  id: string
  titre: string
  consigne: string
  niveau: string
}

Exercice 6.2 — Crée KataMapper dans src/infrastructure/api/mappers/KataMapper.ts

  • KataMapper.toDomain(dto: ApiKataDto): Kata — utilise toKataId() pour convertir l'id
  • KataMapper.toDto(kata: Kata): ApiKataDto — l'inverse
  • C'est une fonction pure — même entrée = même sortie, zéro effet de bord
  • Testable avec Vitest pur — écris les tests avant l'implémentation (TDD)

Question : pourquoi le mapper est dans infrastructure/ et pas dans domain/ ?

Solution — ouvre seulement si bloquée après 20 min 
// src/infrastructure/api/dto/ApiKataDto.ts
export interface ApiKataDto {
  id: string
  titre: string
  consigne: string
  niveau: string
}
// src/infrastructure/api/mappers/KataMapper.ts
import { toKataId } from '@/domain/types/branded'
import type { ApiKataDto } from '../dto/ApiKataDto'
import type { Kata } from '@/domain/types/kata'

export const KataMapper = {
  toDomain(dto: ApiKataDto): Kata {
    return {
      id:          toKataId(dto.id),
      titre:       dto.titre,
      consigne:    dto.consigne,
      niveau:      dto.niveau as Kata['niveau'],
    }
  },

  toDto(kata: Kata): ApiKataDto {
    return {
      id:          kata.id,
      titre:       kata.titre,
      consigne:    kata.consigne,
      niveau:      kata.niveau,
    }
  }
}
// KataMapper.test.ts — écrit en TDD AVANT l'implémentation
describe('KataMapper', () => {
  it('toDomain — convertit l\'id en KataId brandé', () => {
    const dto: ApiKataDto = { id: 'abc-123', titre: 'FizzBuzz', consigne: '...', niveau: 'FACILE' }
    const kata = KataMapper.toDomain(dto)
    expect(kata.id).toBe('abc-123')
    // Le type est vérifié à la compilation : kata.id est KataId, pas string
  })
})

Réponse : le mapper est dans infrastructure/ parce qu'il connaît le format de l'API (un détail d'infrastructure). Le domaine ne devrait pas savoir que les données viennent d'une API HTTP. Si demain tu passes à GraphQL, seul le mapper change.

Questions théoriques

Question 6.1 : pourquoi un DTO d'API ne doit-il pas être réutilisé comme type métier ?

Réponse attendue

Parce qu'il décrit un format d'échange technique, pas une vérité métier stable. Le réutiliser partout couple tout le front au backend et rend chaque évolution beaucoup plus coûteuse.

Question 6.2 : pourquoi un mapper doit-il rester une fonction pure ?

Réponse attendue

Parce qu'on veut pouvoir le tester avec une entrée et une sortie déterministes, sans dépendance externe. Dès qu'il lit un store, un logger ou le réseau, il cesse d'être une simple traduction fiable.

Question 6.3 : quel changement localise un bon Anti-Corruption Layer ?

Réponse attendue

Un changement de contrat externe. Si l'API renomme un champ ou change un format, le coût doit rester concentré dans le DTO et le mapper, pas dans tout le domaine ou tous les composants.

commit suggestion → feat(infra): ajout KataMapper TentativeMapper comme Anti-Corruption Layer

Port KataRepository + adapter

Exercice 7.1 — Interface dans src/domain/ports/KataRepository.ts

findAll(): Promise<Result<Kata[], AppError>> et findById(id: KataId): Promise<Result<Kata, AppError>>. Interface dans domain/ — elle ne sait pas que fetch existe.

Solution — ouvre seulement si bloquée après 20 min 
export interface KataRepository {
  findAll(): Promise<Result<Kata[], AppError>>
  findById(id: KataId): Promise<Result<Kata, AppError>>
}

Exercice 7.2 — ApiKataRepository dans src/infrastructure/api/

  • Implémente KataRepository
  • Utilise KataMapper.toDomain() pour convertir les réponses
  • Mappe les erreurs HTTP vers les AppError typées
  • Ne retourne jamais de throw — toujours un Result
Solution — ouvre seulement si bloquée après 20 min 
export class ApiKataRepository implements KataRepository {
  async findAll(): Promise<Result<Kata[], AppError>> {
    try {
      const res = await fetch(`${API_BASE}/katas`)
      if (!res.ok) return err({ kind: 'network', message: `HTTP ${res.status}` })
      const dtos: ApiKataDto[] = await res.json()
      return ok(dtos.map(KataMapper.toDomain))
    } catch {
      return err({ kind: 'network', message: 'Serveur inaccessible' })
    }
  }
}

Questions théoriques

Question 7.1 : pourquoi le port vit-il dans domain/ alors que son implémentation vit dans infrastructure/ ?

Réponse attendue

Parce que c'est le domaine qui exprime son besoin : "j'ai besoin d'un repository". L'infrastructure ne fait que fournir une réponse concrète à ce besoin.

Question 7.2 : pourquoi retourner un Result depuis le repository plutôt que lancer des exceptions réseau ?

Réponse attendue

Pour garder un contrat homogène et explicite jusqu'aux couches supérieures. Le use case et le store peuvent alors traiter les succès et les erreurs sans dépendre d'un mécanisme d'exception implicite.

Question 7.3 : quel avantage apporte le repository sur un simple fetch dans le composant ?

Réponse attendue

Il centralise l'accès aux données, la conversion DTO → domaine et le mapping d'erreurs. Le composant ne connaît plus l'URL, le format JSON ni la stratégie d'erreur.

commit suggestion → feat(archi): ajout port KataRepository et adapter ApiKataRepository

Use cases purs

Les use cases sont la logique métier pure. Ils ne savent pas que Vue existe, que Pinia existe, que fetch existe. Ils reçoivent leurs dépendances via le constructeur et retournent des Result.

Exercice 8.1 — GetKatasUseCase dans src/domain/usecases/

  • Reçoit KataRepository via le constructeur
  • Méthode execute(): Promise<Result<Kata[], AppError>>
  • Zéro import Vue, zéro import Pinia
  • Écris d'abord le test (TDD) avec un FakeKataRepository que tu crées en 5 lignes

Question : pourquoi injecter KataRepository (l'interface) et pas ApiKataRepository (la classe) ?

Solution — ouvre seulement si bloquée après 20 min 
// FakeKataRepository pour les tests — 5 lignes
class FakeKataRepository implements KataRepository {
  constructor(private readonly katas: Kata[] = []) {}
  async findAll() { return ok(this.katas) }
  async findById(id: KataId) {
    const k = this.katas.find(k => k.id === id)
    return k ? ok(k) : err({ kind: 'not-found', resource: 'Kata' })
  }
}
// GetKatasUseCase.test.ts — TDD, écrit AVANT l'implémentation
describe('GetKatasUseCase', () => {
  it('retourne les katas disponibles', async () => {
    const fakeKata: Kata = { id: toKataId('1'), titre: 'FizzBuzz', niveau: 'FACILE', consigne: '' }
    const useCase = new GetKatasUseCase(new FakeKataRepository([fakeKata]))
    const result = await useCase.execute()
    expect(result.ok).toBe(true)
    if (result.ok) expect(result.value).toHaveLength(1)
  })

  it('propage l\'erreur du repository', async () => {
    const failingRepo: KataRepository = {
      findAll: async () => err({ kind: 'network', message: 'oops' }),
      findById: async () => err({ kind: 'network', message: 'oops' })
    }
    const result = await new GetKatasUseCase(failingRepo).execute()
    expect(result.ok).toBe(false)
  })
})
// GetKatasUseCase.ts
export class GetKatasUseCase {
  constructor(private readonly repo: KataRepository) {}

  async execute(): Promise<Result<Kata[], AppError>> {
    return this.repo.findAll()
  }
}

Réponse : injecter l'interface permet de passer FakeKataRepository en test et ApiKataRepository en prod. Le use case ne sait pas d'où viennent les données. C'est le principe D de SOLID — et la règle de dépendance de l'archi hexagonale.

Questions théoriques

Question 8.2 : qu'est-ce qui fait qu'un use case est "pur" dans ce guide ?

Réponse attendue

Il ne dépend ni de Vue, ni de Pinia, ni d'un adapter concret. Il reçoit ses dépendances par abstraction et renvoie un résultat explicite sans effet de bord UI.

Question 8.3 : pourquoi le fake repository est-il plus pédagogique qu'un mock opaque généré par une librairie ?

Réponse attendue

Parce qu'il montre clairement quel comportement est simulé et reste lisible comme du vrai code métier. Pour apprendre, un faux simple vaut mieux qu'un double de test magique.

commit suggestion → feat(archi): ajout use cases purs GetKatasUseCase StartTentativeUseCase

Stores Pinia comme état seul

Exercice 9.1 — useKataStore dans src/stores/

  • Le store appelle le use case — jamais fetch directement
  • Il stocke le résultat dans un ref<Kata[]>
  • Il expose un getter katasByNiveau(n: Niveau)
  • L'action loadKatas() retourne void — le state est mis à jour, pas retourné
  • L'action utilise le Result pattern : si erreur, elle met à jour un état d'erreur
  • Test avec createPinia() — use case mocké avec un objet qui implémente l'interface

Question : pourquoi le store ne retourne pas le Result directement à l'appelant ?

Solution — ouvre seulement si bloquée après 20 min 
// src/stores/useKataStore.ts
import { defineStore } from 'pinia'
import { ref, readonly, computed } from 'vue'
import type { GetKatasUseCase } from '@/domain/usecases/GetKatasUseCase'

export const useKataStore = defineStore('kata', () => {
  const katas     = ref<Kata[]>([])
  const loadError = ref<string | null>(null)
  const isLoading = ref(false)

  // injecté via provide/inject dans main.ts
  const getKatasUseCase = inject<GetKatasUseCase>('getKatasUseCase')!

  const katasByNiveau = computed(() =>
    (n: Niveau) => katas.value.filter(k => k.niveau === n)
  )

  async function loadKatas() {
    isLoading.value = true
    loadError.value = null
    const result = await getKatasUseCase.execute()
    if (result.ok) {
      katas.value = result.value
    } else {
      loadError.value = toUserMessage(result.error)
    }
    isLoading.value = false
  }

  return {
    katas: readonly(katas),
    loadError: readonly(loadError),
    isLoading: readonly(isLoading),
    katasByNiveau,
    loadKatas,
  }
})

Réponse : le store est une source de vérité partagée. Plusieurs composants peuvent lire katas sans qu'aucun ne re-déclenche le chargement. Si le store retournait le Result, chaque appelant devrait le gérer — on reproduit le problème qu'on essayait de résoudre.

Questions théoriques

Question 9.2 : pourquoi un store n'est-il pas le bon endroit pour faire du mapping DTO → domaine ?

Réponse attendue

Parce que le store doit surtout conserver et exposer l'état. S'il commence à traduire les données ou à parler HTTP, il cumule plusieurs responsabilités.

Question 9.3 : quelle différence entre "source de vérité" et "orchestrateur réseau" ?

Réponse attendue

Une source de vérité centralise l'état partagé. Un orchestrateur réseau gère les appels externes, le mapping et les erreurs techniques. Dans cette architecture, le store fait surtout le premier rôle.

Exercice 10.1 — useAsync dans src/infrastructure/composables/useAsync.ts

  • Signature : useAsync<T>(fn: () => Promise<Result<T, AppError>>)
  • Utilise ref de Vue — pas onMounted ni useRouter
  • Retourne { state: Ref<AsyncState<T>>, execute: () => Promise<void> }
  • Test avec @vue/test-utils minimal (pas besoin de monter un composant)
Solution — ouvre seulement si bloquée après 20 min 
export function useAsync<T>(fn: () => Promise<Result<T, AppError>>) {
  const state = ref<AsyncState<T>>(idle())

  async function execute() {
    state.value = loading()
    const result = await fn()
    state.value = result.ok ? success(result.value) : error(toUserMessage(result.error))
  }

  return { state, execute }
}

Questions théoriques

Question 10.1 : pourquoi ce composable reste-t-il en infrastructure/ alors qu'il utilise ref ?

Réponse attendue

Parce qu'il reste headless : il n'utilise ni cycle de vie, ni router, ni composant. Il est lié à Vue réactivité, mais pas à l'UI affichée.

Question 10.2 : pourquoi encapsuler le boilerplate async dans un composable dédié ?

Réponse attendue

Pour éviter de réécrire partout le même cycle loading → success/error. Cela réduit la duplication et garantit un comportement homogène.

Question 10.3 : pourquoi le composable reçoit-il une fonction, et pas directement une promesse déjà lancée ?

Réponse attendue

Parce qu'on veut contrôler précisément quand l'exécution démarre, pouvoir relancer l'opération et tester le composable sans déclencher d'effet de bord à l'instanciation.

Questions théoriques

Question 12.1 : quel est le rôle exact d'une vue après ce refactoring ?

Réponse attendue

Assembler des composants, brancher un composable UI et afficher l'état courant. La vue décide du rendu, pas de l'accès aux données ni des traductions de format.

Question 12.2 : pourquoi les attributs d'accessibilité role="status" et role="alert" sont-ils mentionnés ici ?

Réponse attendue

Parce qu'une interface fonctionnelle ne se limite pas à "ça s'affiche". Elle doit aussi communiquer correctement les changements d'état aux technologies d'assistance.

Question 12.3 : quel indice te montre qu'une vue est encore trop couplée ?

Réponse attendue

La présence de détails réseau, de mapping technique ou de logique métier dans le fichier de vue. Une vue trop couplée devient vite longue, fragile et difficile à tester.

🚦Sonar quality gate CI≈ contrôle qualité centralisé

Ajouter SonarQube ou SonarCloud comme barrière de qualité bloquante dans la CI, une fois le projet assez dense pour que les métriques et les alertes aient un vrai signal.

🤝Pact / contract testing avancé≈ contrat API côté consommateurs

Introduire les tests de contrat consommateurs/producteurs si le projet évolue vers plusieurs clients, plusieurs services ou des besoins de synchronisation d'équipes plus forts.

🧬Mutation testing PIT≈ vérifier que les tests attrapent vraiment les bugs

Mesurer la robustesse réelle des tests backend en introduisant des mutations automatiques du code et en observant si la suite les détecte.

📱Service Worker / PWA≈ couche offline et cache navigateur

Explorer le hors-ligne, le cache applicatif et les comportements réseau avancés quand le produit lui-même justifiera cet investissement.

♿Accessibilité automatisée≈ garde-fous UX dans les tests

Ajouter des contrôles automatiques plus poussés, par exemple en e2e, quand l'interface sera plus stable et qu'un vrai travail d'accessibilité pourra être suivi dans la durée.

🧠TypeDD explicite≈ conception davantage guidée par les types

Faire du TypeDD un vrai sujet V2, pas un prérequis caché de la V1. L'idée est de partir d'une base déjà saine en TypeScript strict et typage fort pragmatique, puis de montrer comment pousser la modélisation plus loin sur un module ciblé.

🧪Pistes d'introduction du TypeDD≈ nouvelles features ou refactorings ciblés

Deux voies naturelles : l'introduire sur une nouvelle fonctionnalité V2 comme un moteur de progression ou un workflow de feedback IA, ou refactorer une zone existante bien comprise, par exemple les états de session ou la modélisation des erreurs, pour comparer avant/après.

🔁Remplacer l'IA sensei par un modèle open source≈ valider que le port IA isole vraiment le domaine

Prendre l'implémentation V1 basée sur l'API IA sensei par défaut, puis brancher une alternative open source pour vérifier qu'un changement de fournisseur ne force ni refonte du domaine, ni couplage caché dans les use cases. Le but n'est pas d'avoir un meilleur modèle, mais de prouver que l'architecture rend ce remplacement faisable.

🧭Comment l'introduire concrètement≈ progression V2 par module

La V2 peut montrer une montée en puissance progressive : d'abord une feature V1 correcte mais simplement typée, puis une version enrichie où les états deviennent des unions fermées, les transitions illégales deviennent impossibles à représenter, et certaines fonctions n'acceptent plus que des formes déjà validées.

🔬Exemples de sujets V2 compatibles TypeDD≈ bons terrains d'expérimentation

Très bons candidats : moteur de progression avec badges et règles, états de session de kata plus riches, workflow de feedback IA par étapes, modélisation d'erreurs plus fine, ou formulaires multi-étapes où tous les états ne sont pas légalement accessibles depuis n'importe où.
